# Object Storage in Etherion

## Why We Store Files Separately

The Etherion platform handles three categories of data that require different storage strategies. **Structured data**—tenant configurations, user profiles, job histories—lives in PostgreSQL where ACID compliance and relational queries matter. But we also handle **unstructured data**: user-uploaded files, agent artifacts, knowledge base documents, execution traces, and media. These blobs have no place in PostgreSQL.

Here's why. Databases are optimized for rows, not megabytes. Storing a 500 MB PDF directly in PostgreSQL bloats the database, slows backups, fragments indexes, and makes replication painful. You can't stream the file efficiently to a client—you must load it entirely into memory first. Cleanup becomes a garbage collection nightmare. PostgreSQL wasn't designed for this, and forcing it to work as a blob store violates its fundamental assumptions.

## The Three Types of Data We Store

**User Uploads.** When a user attaches a file to a knowledge base, sends a document to an agent, or uploads a training artifact, that file lands in object storage. The database keeps only the metadata: filename, size, upload timestamp, content hash, and the storage path. The actual bytes live elsewhere.

**Agent Artifacts.** As an agent executes a job, it generates outputs—code snippets, analysis documents, execution traces. These artifacts are written to storage during execution. The job record in PostgreSQL references them by key, allowing the user to download them later or the agent to retrieve them on subsequent steps.

**Knowledge Base Documents.** When a knowledge base ingests a document for vector search, the raw document bytes are archived in object storage. Alongside it sits the vector embeddings in PostgreSQL, indexed for semantic search. If a user wants to audit what was ingested, they download the original from storage.

## Why MinIO, Not PostgreSQL or Cloud Storage

PostgreSQL with `bytea` or TOAST tables seems simple at first, but it creates operational debt. Every backup copies every blob. Replication stalls. You can't compress archived blobs cheaply. Recovery is slow. Once you accept that blobs should live outside the database, the next question is: where?

Cloud object storage (Google Cloud Storage, AWS S3, Azure Blob) works, but Etherion is designed for self-hosted, air-gapped deployments on bare metal. Cloud dependencies violate that principle. You also pay egress charges, and data residency becomes complicated.

**MinIO** is S3-compatible, self-hosted object storage written in Go. It's fast, reliable, and designed for deployment on your own infrastructure. It speaks the same API as S3, so code written for MinIO can migrate to AWS if needed. MinIO handles replication, erasure coding, encryption, and lifecycle policies. For a self-hosted platform, it's the natural choice.

## Storage Architecture

The Etherion storage layer is abstracted behind a `StorageBackend` interface. At startup, the system reads the `STORAGE_BACKEND` environment variable and initializes the appropriate implementation. This abstraction allows three deployment models:

**Production.** MinIO is deployed alongside the Etherion backend, exposed on a private network. All file operations go through the MinIO SDK.

**Development.** A local filesystem backend stores files in `/tmp/etherion-storage` by default, making it trivial to inspect files during debugging. URLs are `file://` paths.

**Legacy Migration.** For users migrating from older GCS-backed deployments, a GCS backend is available. New installations should not use it.

The factory automatically selects the backend at startup:

```python
def get_storage_backend() -> StorageBackend:
    """Factory — reads STORAGE_BACKEND env var (local | minio | gcs). Default: minio."""
    backend = os.getenv("STORAGE_BACKEND", "minio").lower()
    if backend == "gcs":
        from .storage_backend_gcs import GCSStorageBackend
        return GCSStorageBackend()
    elif backend == "local":
        from .storage_backend_local import LocalStorageBackend
        return LocalStorageBackend()
    else:
        from .storage_backend_minio import MinIOStorageBackend
        return MinIOStorageBackend()
```

This pattern allows tests to inject a local backend, production to use MinIO, and migrations to happen gradually.

## Bucket Naming and Organization

Data is organized into three main buckets:

- **etherion-artifacts**: Job outputs, execution traces, code generated by agents, temporary working files
- **etherion-kb**: Knowledge base documents, vector embeddings exports, ingestion logs
- **etherion-media**: User profile pictures, team logos, miscellaneous media assets

Within each bucket, files are organized by prefix. For example, a knowledge base document might be stored as `kb-source-docs/tenant-abc/doc-123/document.pdf`. This hierarchical organization allows efficient listing and cleanup via prefix operations.

## The Interface

Every storage backend implements the same seven methods. This is the contract that code depends on:

```python
class StorageBackend(ABC):
    @abstractmethod
    def upload(self, bucket: str, key: str, data: BinaryIO,
               content_type: str = "application/octet-stream") -> str:
        """Upload data and return the object URI."""

    @abstractmethod
    def download(self, bucket: str, key: str) -> bytes:
        """Download and return object bytes."""

    @abstractmethod
    def delete(self, bucket: str, key: str) -> None:
        """Delete an object."""

    @abstractmethod
    def exists(self, bucket: str, key: str) -> bool:
        """Return True if the object exists."""

    @abstractmethod
    def list_keys(self, bucket: str, prefix: str = "") -> list[str]:
        """List object keys under prefix."""

    @abstractmethod
    def get_url(self, bucket: str, key: str, expiry_seconds: int = 3600) -> str:
        """Return a (possibly pre-signed) URL for the object."""

    @abstractmethod
    def health_check(self) -> bool:
        """Return True if the backend is reachable."""
```

Each backend returns a storage URI: MinIO returns `s3://bucket/key`, local returns `file:///path`, GCS returns `gs://bucket/key`. This URI is stored in PostgreSQL, allowing the system to retrieve the file later regardless of the backend implementation.
