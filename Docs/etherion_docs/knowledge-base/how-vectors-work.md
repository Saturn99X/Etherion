# How Vectors Work: Embeddings and Semantic Search

## What Is a Vector?

At its core, a vector is just a list of numbers. In Etherion's knowledge base, each document is represented as a vector with 1536 dimensions (configurable via `KB_EMBEDDING_DIM` environment variable). Think of this as a point in a 1536-dimensional space — far too many dimensions to visualize, but the math works perfectly.

```
Example embedding vector (truncated):
[0.0234, -0.145, 0.892, -0.012, 0.456, ... (1532 more values) ... 0.234]
```

These numbers are generated by an embedding model that has learned to represent text as vectors through neural network training. The key insight is that this encoding is *semantic* — the model has learned that similar concepts should map to similar numerical values.

## Why Similar Text Has Similar Vectors

Imagine you trained a model on millions of text examples. During training, the model learns that words and phrases with similar meanings should produce vectors that are geometrically close to each other in this high-dimensional space.

For example:
- The text "user cannot log in" might embed as `[0.1, 0.8, -0.2, ...]`
- The text "authentication failure" might embed as `[0.12, 0.78, -0.19, ...]`

Notice the numbers are very close — not identical, but similar. This is by design. The model learned that both phrases are about login problems, so they should cluster together in vector space.

In contrast:
- The text "what's the weather today?" might embed as `[-0.7, 0.1, 0.5, ...]`

This vector is very different from the login-related ones because weather is a completely different topic.

## Cosine Similarity: Measuring Closeness

When you want to find documents similar to a query, you need a way to measure how close two vectors are. The standard metric is **cosine similarity**, which measures the angle between two vectors.

Two vectors pointing in the same direction (similar concepts) have a cosine similarity close to 1.0. Two vectors pointing opposite directions have similarity close to -1.0. Orthogonal vectors (no relation) have similarity close to 0.

```
Query vector:     [0.11, 0.79, -0.21, ...]  (user login issue)
Doc 1 vector:     [0.10, 0.80, -0.20, ...]  (authentication error)
  → Cosine similarity: 0.987 (very similar)

Doc 2 vector:     [-0.69, 0.09, 0.51, ...]  (weather question)
  → Cosine similarity: 0.024 (completely different)
```

This is why pgvector uses the `<=>` operator to compute cosine distance (1 - cosine similarity) — it's efficient and mathematically sound for semantic search.

## Concrete Example: Document Retrieval

Let's walk through how Etherion retrieves documents:

1. **Ingestion time**: Your team uploads a PDF titled "Fixing Login Errors". Etherion extracts the text and splits it into chunks:
   ```
   Chunk 1: "Users who see 'Invalid credentials' should reset their password..."
   Chunk 2: "If you can't remember your password, use the recovery link..."
   ```
   Each chunk is embedded independently. Chunk 1 embeds as `[0.105, 0.812, -0.198, ...]`, Chunk 2 as `[0.089, 0.791, -0.211, ...]`.

2. **Query time**: An agent searches for "I can't log into my account". This query string is embedded using the same model:
   ```
   Query: "I can't log into my account"
   Embedding: [0.108, 0.815, -0.195, ...]
   ```

3. **Search**: PostgreSQL computes cosine similarity between the query vector and every document vector in that tenant's knowledge base:
   - Query vs Chunk 1: similarity ≈ 0.991
   - Query vs Chunk 2: similarity ≈ 0.983
   - Query vs "What's the weather today?": similarity ≈ 0.021

4. **Ranking**: Results are sorted by similarity (highest first) and returned to the agent. The agent gets the login-related chunks because they semantically match, even though the wording is different.

## How pgvector Stores and Queries Vectors

PostgreSQL with the pgvector extension stores vectors as a native column type. In Etherion's schema, the `Document` table has an `embedding` column:

```python
from sqlalchemy import Column, Text
from pgvector.sqlalchemy import Vector
import os

def _embedding_column():
    dim = int(os.getenv("KB_EMBEDDING_DIM", "1536"))
    return Column(Vector(dim), nullable=True)

class Document(SQLModel, table=True):
    id: Optional[int] = Field(default=None, primary_key=True)
    doc_id: str = Field(unique=True, index=True, nullable=False)
    text_chunk: str = Field(sa_column=Column(Text, nullable=False))
    embedding: Optional[Any] = Field(default=None, sa_column=_embedding_column())
    metadata_json: Optional[str] = Field(default=None)
    # ... tenant_id, created_at, etc.
```

When querying, you convert the query embedding to a string and use the `<=>` operator (cosine distance):

```sql
SELECT d.doc_id, d.text_chunk, d.metadata_json,
       1 - (d.embedding <=> CAST(:vec AS vector)) AS score
FROM document d
WHERE d.tenant_id = :tenant_id
ORDER BY score DESC
LIMIT :top_k
```

Breaking this down:
- `CAST(:vec AS vector)` converts the query embedding string to pgvector's vector type
- `d.embedding <=> CAST(:vec AS vector)` computes cosine distance (0 = identical, 2 = opposite)
- `1 - (...)` converts distance to similarity (1 = identical, 0 = opposite, -1 = complete opposite)
- `ORDER BY score DESC` puts highest similarity first

pgvector supports HNSW (Hierarchical Navigable Small World) indexes for speed:

```sql
CREATE INDEX ON document USING hnsw (embedding vector_cosine_ops)
```

With this index, even searching across millions of documents is fast because pgvector doesn't compare every vector — it uses a proximity graph to explore nearby neighborhoods.

## Why Embeddings Matter for Agents

Agents need context to reason effectively. Vector search provides *relevant* context by finding documents that semantically match the problem, not just keyword-matching. This means:

- An agent searching for "authentication failure" retrieves documents about password resets, OAuth tokens, session timeouts — all conceptually related.
- If a document says "users reported that their login stopped working", it still gets retrieved because semantically it's about the same problem, even with completely different wording.

This semantic understanding elevates agent responses from keyword-based to truly contextual reasoning.
