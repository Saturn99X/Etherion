# Tool Dispatch: How Tools Work and Approval Flows

Tools are the actions agents can take—search the knowledge base, query a database, send an email, etc. This document explains the tool system, how tools are registered and invoked, and how approval works for sensitive operations.

## The Tool Registry

All tools in Etherion are registered in a central **Tool Registry**. The registry is the source of truth for what tools exist and their metadata.

```python
# From src/database/models/tool.py
class ToolStatus(str, Enum):
    """Tool status enumeration for security and lifecycle management."""
    STABLE = "STABLE"      # Production-ready
    BETA = "BETA"          # Functional but untested in production
    DEPRECATED = "DEPRECATED"  # End-of-life

class Tool(SQLModel, table=True):
    """Tool metadata stored in the database."""
    id: Optional[int] = Field(default=None, primary_key=True)
    name: str = Field(unique=True, index=True)  # Unique tool identifier
    description: str                            # What it does
    status: ToolStatus = Field(default=ToolStatus.BETA)
    is_custom_agent_executor: bool = False      # Is this a meta-tool?
    version: Optional[str] = None               # Tool version
    documentation_url: Optional[str] = None     # Help link
    category: Optional[str] = None              # Grouping (e.g., "research")
    requires_auth: bool = False                 # Auth requirement
    max_concurrent_calls: Optional[int] = None  # Rate limiting
    created_at: datetime                        # Tracking
    last_updated_at: datetime
```

When a tenant creates an agent team or specialist, they reference tools by name (strings). At runtime, the system looks up the tool definition and loads its implementation.

## What a LangChain Tool Looks Like

Etherion uses LangChain's tool abstraction. A tool is a callable (function or class) with:

1. **A name** — Unique identifier (e.g., "kb_search")
2. **A description** — What it does (read by LLMs)
3. **An input schema** — What arguments it accepts (JSON schema)
4. **An implementation** — The actual logic

Here's an example:

```python
# Example: Knowledge Base Search Tool
from langchain.tools import tool
import json

@tool
def kb_search(query: str, limit: int = 5) -> str:
    """
    Search the tenant's knowledge base for documents matching the query.

    Args:
        query: The search query (natural language or keywords).
        limit: Maximum number of results to return.

    Returns:
        JSON string with matching documents and their content.
    """
    # Pseudocode
    results = kb_service.search(query, limit=limit)
    return json.dumps([
        {
            "doc_id": doc.id,
            "title": doc.title,
            "excerpt": doc.content[:500],
            "relevance_score": score,
        }
        for doc, score in results
    ])

# Tool schema (auto-generated by LangChain)
kb_search.name        # "kb_search"
kb_search.description # Docstring
kb_search.args        # {"query": {...}, "limit": {...}}
```

When an LLM (like Gemini) uses this tool, it sees:

```json
{
  "name": "kb_search",
  "description": "Search the tenant's knowledge base...",
  "parameters": {
    "type": "object",
    "properties": {
      "query": {"type": "string", "description": "The search query"},
      "limit": {"type": "integer", "description": "Max results"}
    },
    "required": ["query"]
  }
}
```

## Tool Invocation Flow

Here's how a tool gets called when a specialist needs it:

### Step 1: Parse Tool Call

The specialist's LLM returns a message like:

```
I'll search the knowledge base for sales data.

Tool: kb_search
Arguments: {"query": "Q3 revenue trends", "limit": 10}
```

The specialist executor parses this into:

```python
tool_call = {
    "name": "kb_search",
    "arguments": {
        "query": "Q3 revenue trends",
        "limit": 10,
    }
}
```

### Step 2: Check Pre-Approval

Before invoking, verify the tool is in the allowed list:

```python
# From specialist_executor.py (pseudocode)
tool_name = tool_call['name']
allowed_tools = specialist.get_tool_names()  # JSON list from DB

if tool_name not in allowed_tools:
    error_message = f"Tool '{tool_name}' is not allowed for this specialist."
    logger.warning(f"Tool not approved: {tool_name}")
    # Append error to LLM conversation and retry
    response += f"\n[ERROR] {error_message}"
    response = await llm.ainvoke(prompt + response)
else:
    # Proceed to invocation
```

This is the **first security gate**. A specialist cannot call a tool just because it exists; it must be explicitly on the specialist's `tool_names` list.

### Step 3: Load Tool Implementation

Look up the tool by name and get its implementation:

```python
# Pseudocode: Tool Loader
def get_tool_by_name(tool_name: str) -> Callable:
    """Get the tool implementation by name."""
    # Check if it's a pre-built tool
    if tool_name == "kb_search":
        return kb_search_tool()
    elif tool_name == "sql_query":
        return sql_query_tool()
    elif tool_name == "send_email":
        return send_email_tool()
    else:
        raise ValueError(f"Tool '{tool_name}' not found in registry")

tool_impl = get_tool_by_name(tool_call['name'])
```

### Step 4: Check Tool Status and Auth

Before calling, check if the tool is available:

```python
# Pseudocode: Pre-invocation Checks
tool_record = session.query(Tool).filter(Tool.name == tool_name).first()

if not tool_record:
    raise ToolNotFound(f"Tool '{tool_name}' not registered")

if tool_record.status == ToolStatus.DEPRECATED:
    raise ToolDeprecated(f"Tool '{tool_name}' is deprecated; use {tool_record.replacement}")

if tool_record.status == ToolStatus.BETA:
    logger.warning(f"Tool '{tool_name}' is in BETA; results may be unstable")

if tool_record.requires_auth:
    # Check if tenant has credentials
    creds = session.query(ToolCredential).filter(
        ToolCredential.tenant_id == tenant_id,
        ToolCredential.tool_name == tool_name,
    ).first()
    if not creds or not creds.is_valid():
        raise ToolAuthRequired(f"Tool '{tool_name}' requires authentication")
```

### Step 5: Invoke the Tool

Now call it:

```python
# Pseudocode: Tool Invocation
try:
    result = await tool_impl.ainvoke(tool_call['arguments'])
    logger.info(f"Tool '{tool_name}' returned: {result}")
except Exception as e:
    logger.error(f"Tool '{tool_name}' failed: {e}")
    result = f"[ERROR] {str(e)}"

# Append result to specialist's conversation
response += f"\n[TOOL RESULT] {tool_name}: {result}"
```

### Step 6: Feed Result Back

The specialist's LLM sees the result and continues reasoning:

```
[TOOL RESULT] kb_search: [
  {"doc_id": "sales_q3_report", "title": "Q3 Sales Summary", "excerpt": "Revenue: $2.5M..."},
  {"doc_id": "market_analysis", "title": "Market Trends", "excerpt": "Competition up 15%..."}
]

Great! I found two relevant documents. Let me analyze the data...
[Continues reasoning with tool results]
```

## Tool Approval: Pre-Approved vs. Manual Review

The approval strategy depends on the tool and the operation:

### Pre-Approved Tools

Some tools are always allowed (if they're in the team's pre-approved list). Example:

```python
# From orchestrator_runtime.py
tool_policy=ToolApprovalPolicy(
    auto_approved_tools=[
        "search_personal_kb",
        "search_project_kb",
        "confirm_action_tool",
    ],
    manual_review_states=["submitted", "pending_platform_orchestrator"],
    required_reviewers=["platform_orchestrator"],
    confirmation_prompt_text=(
        "The team orchestrator requires approval to use {tool_name}. "
        "Provide justification: {justification}"
    ),
    fallback_behavior="escalate_to_platform",
)
```

If a specialist calls `search_personal_kb` and it's in the team's `pre_approved_tool_names`, it runs immediately.

### Manual Review (Human-in-the-Loop)

For sensitive tools (e.g., "send_email", "delete_document"), the call is queued for approval:

```python
# From tool_request_queue.py
from src.services.tool_request_queue import ToolRequestQueue

queue = ToolRequestQueue()

# 1. Enqueue the request
request_id = queue.enqueue(
    job_id="job_abc123",
    tool_name="send_email",
    args={
        "recipient": "customer@example.com",
        "subject": "Special Offer",
        "body": "...",
    }
)
logger.info(f"Queued tool request: {request_id}")

# 2. Wait for approval (with timeout)
approved = queue.wait_for_approval(request_id, timeout=300)  # 5 min

if approved:
    # Execute the tool
    result = await send_email_tool.ainvoke(tool_call['arguments'])
else:
    # Rejection
    result = "[REJECTED] Tool call was not approved"
```

The request lives in Redis and is delivered to a human reviewer (via the dashboard or API). The reviewer can approve or reject it:

```python
# Pseudocode: Approval Handler (human via dashboard)
if user_clicked_approve_button:
    queue.approve(request_id)
elif user_clicked_reject_button:
    queue.reject(request_id)
```

## The Tool Request Queue (Redis-Backed)

The approval mechanism uses Redis for low-latency, async communication:

```python
# From src/services/tool_request_queue.py
class ToolRequestQueue:
    _PREFIX = "etherion:toolreq"
    _APPROVAL_PREFIX = "etherion:toolapproval"

    def enqueue(self, job_id: str, tool_name: str, args: Dict[str, Any]) -> str:
        """Enqueue a tool call for approval."""
        request_id = f"tr_{uuid.uuid4().hex[:12]}"
        payload = json.dumps({
            "request_id": request_id,
            "job_id": job_id,
            "tool_name": tool_name,
            "args": args,
        })
        # Store in Redis under job-specific key
        self._client().lpush(f"{_PREFIX}:{job_id}", payload)
        return request_id

    def wait_for_approval(self, request_id: str, timeout: int = 300) -> bool:
        """Block until approval/rejection arrives (with timeout)."""
        result = self._client().brpop(
            f"{_APPROVAL_PREFIX}:{request_id}",
            timeout=timeout
        )
        if result is None:
            return False  # Timeout
        _, value = result
        return json.loads(value).get("approved", False)

    def approve(self, request_id: str) -> None:
        """Approve a queued request."""
        self._client().lpush(
            f"{_APPROVAL_PREFIX}:{request_id}",
            json.dumps({"approved": True}),
        )

    def reject(self, request_id: str) -> None:
        """Reject a queued request."""
        self._client().lpush(
            f"{_APPROVAL_PREFIX}:{request_id}",
            json.dumps({"approved": False}),
        )
```

The queuing uses Redis lists and blocking pops (`brpop`), so:
- Multiple tool requests can be pending.
- Approvers don't need to poll; Redis notifications push updates.
- Timeout is enforced; if no approval after 5 minutes, the tool call fails.

## Example: Complete Tool Call with Approval

Let's trace a realistic scenario: a specialist wants to send a customer email.

**Setup:**
- Specialist: Sales Outreach Agent
- Team pre-approved tools: `["kb_search", "email_template_finder"]`
- Sensitive tool: `"send_email"` (not pre-approved)

**Execution:**

```
1. Specialist decides: "I'll send an email to customer@acme.com"
   Tool: send_email
   Args: {"to": "customer@acme.com", "template": "upsell_offer"}

2. Specialist executor checks: Is "send_email" in allowed tools?
   No. Check tool policy.

3. Is "send_email" auto-approved? No.
   → Enter manual review flow.

4. Enqueue request:
   request_id = queue.enqueue(
       job_id="job_xyz",
       tool_name="send_email",
       args={"to": "customer@acme.com", "template": "upsell_offer"}
   )
   → request_id = "tr_abc123def456"

5. Specialist executor waits:
   approved = queue.wait_for_approval("tr_abc123def456", timeout=300)
   → Blocking...

6. Meanwhile, a human reviewer sees the pending request in the dashboard:
   "Specialist wants to send email to customer@acme.com. Approve?"

7. Human clicks "Approve":
   queue.approve("tr_abc123def456")

8. Redis unblocks the wait_for_approval call:
   approved = True

9. Specialist executor continues:
   if approved:
       result = await send_email_tool.ainvoke({"to": "...", "template": "..."})
       response += f"[TOOL RESULT] send_email: Email sent successfully"
   else:
       response += "[REJECTED] Email send was not approved by human reviewer"

10. Specialist continues reasoning with the result.
```

If the human clicks "Reject" instead:
```
7. Human clicks "Reject":
   queue.reject("tr_abc123def456")

8. approved = False

9. Specialist executor sees rejection and either:
   - Informs the user of the rejection
   - Tries a different approach
   - Reports the failure to the orchestrator
```

## Cost Tracking for Tools

Tools may incur costs (e.g., API calls to external services). The cost is tracked:

```python
# Pseudocode: Tool Cost Tracking
cost_tracker = CostTracker(job_id=job_id, tenant_id=tenant_id)

# Some tools are metered
tool_record = session.query(Tool).filter(Tool.name == tool_name).first()
if tool_record.cost_per_call:
    cost_tracker.add_tool_call(tool_name, cost=tool_record.cost_per_call)

# Check budget before invoking
if cost_tracker.total_cost() > max_budget:
    raise CostGuardrailExceeded("Tool call would exceed budget")

# Invoke and track
result = await tool_impl.ainvoke(tool_call['arguments'])
```

## Tool Instrumention and Observability

All tool calls are instrumented for observability:

```python
# From src/services/tool_instrumentation.py (pseudocode)
def instrument_tool(tool_func: Callable, tool_name: str) -> Callable:
    """Wrap a tool to add logging and tracing."""
    async def wrapper(input_dict: Dict[str, Any]) -> str:
        start_time = time.time()
        try:
            result = await tool_func(input_dict)
            elapsed = time.time() - start_time
            logger.info(f"Tool {tool_name} completed in {elapsed:.2f}s")
            return result
        except Exception as e:
            elapsed = time.time() - start_time
            logger.error(f"Tool {tool_name} failed after {elapsed:.2f}s: {e}")
            raise
    return wrapper

# Tools are wrapped before passing to specialists
kb_search_instrumented = instrument_tool(kb_search, "kb_search")
```

## Tool Integration: From Approval to Result

Here's the complete picture:

```
Specialist LLM decides to call a tool
    ↓
Tool call parsed from LLM response
    ↓
Check if tool in specialist's allowed_tools
    ├─ No? Reject, retry with error
    └─ Yes? Continue
    ↓
Look up tool in Tool registry
    ├─ Not found? Reject
    ├─ Deprecated? Reject
    └─ Available? Continue
    ↓
Check tool pre-approval policy
    ├─ Auto-approved? Invoke immediately
    ├─ Manual review? Queue for approval
    │   ├─ Human approves? Invoke
    │   ├─ Human rejects? Fail
    │   └─ Timeout? Fail
    └─ Continue
    ↓
Load tool implementation
    ↓
Track cost
    ↓
Invoke tool with arguments
    ↓
Result appended to specialist's conversation
    ↓
Specialist continues reasoning
```

## Summary

- **Tool Registry**: Central source of truth (database) with metadata and status.
- **LangChain Tools**: Callable with name, description, schema, and implementation.
- **Pre-Approval**: Specialists can only call tools in their `tool_names` list.
- **Manual Approval**: Sensitive tools are queued in Redis; humans approve/reject.
- **Cost Tracking**: Tools may incur costs; guardrails prevent overspend.
- **Observability**: All tool calls are logged and traced for audit.
