## 1. Platform Overview and System Topology

Etherion is a self-hosted multi-tenant AI orchestration platform that enables organisations to build, deploy, and run structured workflows powered by teams of specialised AI agents. Rather than asking users to manage individual language models or write custom orchestration logic, Etherion provides a unified platform where multiple tenants (organisations) can define projects containing agent teams that collaborate to solve complex tasks. Each tenant operates within an isolated boundary: their own users, projects, agents, and data remain completely segregated in the system. The platform ships as installable Python packages (via pip) accompanied by a CLI for initial setup and a terminal-based UI for ongoing management, making it accessible both for development environments and production self-hosted deployments.

The core problem Etherion solves is the operational complexity of AI agent orchestration. Building production-grade AI systems today requires coordinating multiple services—orchestration engines, LLMs, storage, vectorization, job scheduling, and more—along with multi-tenancy, security, and state management. Etherion abstracts this complexity into a cohesive platform: a user defines an agent team and a workflow structure through the UI or API, submits a job with parameters, subscribes to live progress updates, and receives results and artefacts when complete. All the infrastructure concerns—async task queuing, vector search, credential isolation, rate limiting, audit logging—operate transparently beneath a simple request-response contract.

Etherion's architecture consists of five core services that work together to implement this model. The **FastAPI + Strawberry GraphQL API server** (running via uvicorn on port 8080 by default) is the primary entry point; it exposes a GraphQL API for all client operations and handles WebSocket subscriptions for real-time updates. The **Celery worker pool** (with Redis as broker, configured for 8 concurrent workers and prefetch=1) processes asynchronous jobs enqueued by API mutations; workers execute the actual agent orchestration logic, which may invoke external tools, run LLM chains, and generate artefacts. **PostgreSQL with the pgvector extension** serves as the persistent data store, storing tenant metadata, users, projects, agent definitions, job records, and vector embeddings for semantic search across knowledge bases. **Redis** plays a dual role: it acts as the Celery job broker and message bus for low-latency pub/sub, which powers the subscription mechanism that streams job progress events back to connected clients in real time. Finally, **MinIO** (object storage compatible with S3) holds all unstructured data—uploaded files, knowledge base documents, artefacts generated by agent runs, and other binary assets. Each tenant has isolated storage buckets within MinIO, ensuring data residency boundaries are enforced at the storage layer.

The request lifecycle for an AI job demonstrates how these services coordinate. A client sends a GraphQL mutation to the API (for example, `createAgentRun`) with parameters and authentication headers. The request passes through a middleware stack that applies CORS, security headers, rate limiting, CSRF validation, and tenant context extraction from the JWT. The tenant middleware resolves the tenant ID and stores it in request state, ensuring all downstream database queries are scoped to that tenant's data. The GraphQL resolver validates inputs, persists a job record to PostgreSQL, then enqueues a Celery task to the worker pool, returning a `job_id` to the client immediately. The client then opens a GraphQL subscription connection (via WebSocket) to `/graphql`, passing the same authentication headers and the `job_id`. This subscription taps into a Redis pub/sub channel specific to that job. Meanwhile, the Celery worker dequeues the task, loads the job and agent definitions from the database, executes the orchestration pipeline (which may chain LLM calls, invoke tools, perform vector search against MinIO-backed documents, and store intermediate results), and publishes progress events to Redis at key milestones—e.g., "agent_started", "tool_invoked", "step_completed". The Redis pub/sub layer delivers these events to all subscribed clients in near real time. When the job completes, the worker stores the final result and any generated artefacts in PostgreSQL and MinIO, publishes a "job_completed" event to Redis, and the client subscription receives it and terminates. The entire flow provides the user with a streaming view of agent execution while the backend manages complexity: multi-step workflows, credential rotation, cost tracking, concurrency control, and error recovery are all abstracted away.

## 2. Data Model and Multi-Tenancy

### The Data Model

Etherion stores all application state in a PostgreSQL database organized around a core hierarchy of ownership and relationships. At the root is the **Tenant**, which represents a complete isolated workspace: a company, organization, or team using the platform. Each tenant has an internal numeric `id` and a user-facing string identifier called `tenant_id` (13 characters, URL-safe, like `t_abc123xyz`). Tenants also store a unique `subdomain` for invite flows and onboarding, plus administrative metadata like the default data retention policy in days.

**Users** belong to tenants and authenticate either via OAuth providers (Google, Apple) or local password-based accounts. Each user has a `user_id` external identifier (from the OAuth provider or generated), email, display name, and permission flags (`is_admin`, `is_active`). Users may own multiple **Projects**, which are workspace containers for agents, files, and conversations. Projects do not have owners in the authorization sense—any user in the tenant can access all projects—but they are linked to the user who created them for lineage and optional filtering.

Within each project, **Conversations** represent distinct dialogue sessions with the AI agents. Conversations contain **Messages** organized in sequence (role: user/assistant/system, content, timestamp). This design allows the same user or team to run multiple independent agent conversations within a project.

Projects can also store **ProjectKBFiles** (knowledge base files), which are typically ingested documents, code snippets, or reference materials. These files have a `status` field (processing, available, or failed), file size, MIME type, and an optional error message if ingestion fails. Files respect a configurable retention policy in days and can be automatically archived or deleted. Each file is stored in object storage (MinIO) and tracked in the database.

Behind the scenes, **ExecutionCost** records track token usage and costs per job step: which step name (e.g., "Specialist: drafting_agent"), which model was used, input/output token counts, and the USD cost. This allows billing and analytics per tenant. **Job** records capture long-running async tasks with their status (pending, running, completed, failed) and results, also scoped to tenant and user.

**CustomAgentDefinition** and **AgentTeam** entities allow tenants to define bespoke AI agent configurations and organize them into named groups with approved tools. Both are tenant-scoped; users on different tenants cannot see or modify each other's agent definitions.

**UserObservation** records capture learned behavioral patterns about individual users: preferred tone, technical level, decision-making style, successful and failed approaches, and even stress patterns and frustration triggers. These observations, accumulated over multiple interactions, inform the system on how best to tailor responses and tool usage for each user.

Supporting entities include **ToneProfile** (per-user communication style templates), **Expense** (records of vendor transactions for analysis), **CreditLedger** and **TenantCreditBalance** (payment and credit tracking), **TenantInvite** (time-limited tokens for adding new team members), and **IPAddressUsage** (optional rate-limiting on signup and tenant creation).

Every single table carries a `tenant_id` foreign key. This design choice is both a blessing and a safeguard: it ensures the database schema is tenant-aware at every layer.

### Multi-Tenancy and Row-Level Security

Etherion enforces multi-tenancy at the database layer using PostgreSQL Row-Level Security (RLS) policies. The model is simple but powerful: before executing any application query, the middleware extracts the current user's tenant ID and executes a single SQL command on the database connection:

```
SET app.tenant_id = <numeric_tenant_id>
```

This sets a session-level configuration variable that PostgreSQL RLS policies can read. Then, when the application executes a query—even a bare `SELECT *` with no WHERE clause—PostgreSQL's RLS engine automatically filters the result set to rows where the `tenant_id` column matches the session variable. If the session variable is not set or is NULL, rows with non-NULL `tenant_id` are excluded (useful during onboarding flows when the tenant is not yet known).

**Why RLS matters:** Even if the application code forgets to add a `WHERE tenant_id = ?` clause to a query, or if a developer inadvertently creates a bug that leaks a database session, the database itself refuses to return or modify rows outside the current tenant. This is a defense-in-depth strategy. The application still filters explicitly (in ORM queries and GraphQL resolvers), but the database is the final gate.

The RLS enforcement happens transparently:
1. A request arrives with an Authorization header (Bearer token).
2. The security middleware decodes the token to extract `user_id` and `tenant_id`.
3. Before the request's code can touch the database, middleware or the service layer calls `set_session_tenant_context(session, tenant_id)`, which executes the `SET app.tenant_id` command on the PostgreSQL connection (using `session.connection()` to ensure it runs on the same connection that will handle subsequent queries).
4. All subsequent SELECT/INSERT/UPDATE/DELETE operations on that connection are filtered or blocked by RLS.
5. On exception or connection reuse (checkout), the middleware re-applies the tenant context to ensure the right context is always active.

For async workloads, the same pattern applies via `set_session_tenant_context_async()`, and event listeners on the async engine (`connect` and `checkout` events) automatically manage the context variable when connections are created or reused from the pool.

**Onboarding special case:** During tenant signup or user invite acceptance, the tenant ID may not yet be known. In this scenario, the middleware sets `app.tenant_id` to NULL and uses an alternative identifier in RLS policies—typically `app.requested_subdomain` from the request. RLS policies check for this pattern to allow the signup flow to create the tenant and first user without a pre-existing tenant ID.

### Tenant Identity: The Two-Key System

Each tenant has two distinct identifiers, serving different purposes:

- **`id` (numeric primary key):** Internal, used in all foreign key relationships and in the `app.tenant_id` session variable. This is what the RLS policies check against. It is never exposed in public APIs.

- **`tenant_id` (13-character string):** Public, URL-safe, and immutable. Used in API endpoints, GraphQL queries, and anywhere tenants are referenced externally. Generated as a random combination of letters and digits. This identifier is human-readable in logs and error messages but cryptographically unpredictable.

The dual-key approach allows the database schema to remain efficient (numeric FKs are smaller and faster for joins) while keeping public APIs human-friendly and preventing enumeration attacks (an attacker cannot guess a tenant ID by incrementing a numeric value).

Additionally, each tenant has a `subdomain` field, which is unique and used during invite flows. When a user clicks an invite link, the system can look up the tenant by subdomain and allow them to complete signup even before they have been issued a bearer token.

## 3. Authentication and Authorization

### 3.1 Overview

Etherion AI provides a secure, multi-tenant authentication system that supports two sign-in paths: OAuth (Google, GitHub, Microsoft, Slack) and traditional email/password authentication. All authentication flows result in a short-lived JWT token that clients use to establish their identity and tenant context on subsequent requests.

The platform uses **Row-Level Security (RLS)** in PostgreSQL to enforce tenant isolation, meaning the database itself prevents users from accessing data outside their tenant. This is complemented by application-layer authentication middleware that validates tokens, loads user records, and establishes tenant context on each request.

### 3.2 The Two Sign-In Paths

#### 3.2.1 OAuth Path (Google, GitHub, Microsoft, Slack)

The OAuth flow is designed for users who want to authenticate using their existing identities from identity providers. This path is stateless and leverages industry-standard OAuth 2.0 and OpenID Connect protocols.

**How it works:**

1. **Authorization Request**: The client directs the user to the provider's authorization endpoint (e.g., `https://accounts.google.com/o/oauth2/v2/auth`). The user logs in and grants Etherion AI permission to access their basic profile information.

2. **Authorization Code Callback**: The provider redirects the browser back to Etherion AI with an authorization code. The client captures this code and sends it to the backend via a callback endpoint (e.g., `POST /api/auth/callback/google`).

3. **Token Exchange**: The backend exchanges the authorization code for an access token by calling the provider's token endpoint with the client credentials (OAuth application ID and secret). These credentials may be globally configured or tenant-specific (stored in the secrets manager for that tenant).

4. **User Info Retrieval**: Using the provider's access token, the backend fetches the user's profile information (email, name, profile picture) from the provider's userinfo endpoint.

5. **User Record Creation or Update**: The backend checks if a user with this provider ID already exists in the database. If yes, it updates the user's information and last-login timestamp. If no, it creates a new user record.

6. **Tenant Assignment**: When creating a new user in multi-tenant mode, the system automatically creates a new tenant for that user (unless an invite token is provided, in which case the user joins the invited tenant). For existing users, their existing tenant is used.

7. **JWT Issuance**: The backend creates a short-lived JWT token (valid for 30 minutes) containing the user's ID, email, and tenant ID, signs it with the server's `JWT_SECRET_KEY` using HS256 (HMAC-SHA256), and returns it to the client.

**Supported Providers**: Google, GitHub, Microsoft (with configurable tenant), and Slack. Apple OAuth is currently disabled.

#### 3.2.2 Password Path (Email/Password Authentication)

The password path allows users to sign up or log in using their email address and a password. This path is useful for self-hosted deployments or organizations that prefer not to use external identity providers.

**Sign-Up Flow (`POST /api/auth/signup`)**:

1. **Validation**: The endpoint validates that the email is not already registered and applies optional security checks:
   - If `BLOCK_VPN_SIGNUP` is enabled, it queries a VPN detection service to block signup from known hosting providers.
   - If `ENFORCE_SIGNUP_IP_LIMIT` is enabled, it tracks the hashed IP address and limits one account per IP.

2. **Subdomain Selection**: For new accounts, the user must specify a unique subdomain (e.g., `acme-corp`). The backend validates the subdomain format and checks uniqueness against the database. This subdomain becomes the tenant's identifier in URLs.

3. **Tenant Creation**: The system creates a new tenant with the chosen subdomain and sets the signup email as the admin email.

4. **User Creation**: A new user record is created with the email, password hash, name, and tenant ID. The password is hashed using PBKDF2-SHA256 with 200,000 iterations (a deliberate, computationally expensive algorithm that resists brute-force attacks).

5. **Password Hash Format**: Passwords are stored in a custom format: `pbkdf2_sha256$<iterations>$<salt_b64>$<hash_b64>`. For example: `pbkdf2_sha256$200000$Aa+VJDxw1A==$dF3j...`. The salt is 16 random bytes, generated fresh for each password.

6. **Credit Grant**: The new user is automatically granted an initial 100 credits.

7. **JWT Issuance**: A JWT token is created and returned along with the user information and tenant subdomain.

**Login Flow (`POST /api/auth/login`)**:

1. **Email Lookup**: The backend retrieves the user record by email.

2. **Password Verification**: The stored password hash is split on `$` to extract the iteration count, salt, and expected hash. The provided password is hashed using the same algorithm and parameters. The computed hash and expected hash are compared using constant-time comparison (`hmac.compare_digest`) to prevent timing attacks.

3. **Tenant Enforcement**: If an invite token is provided during login, the system verifies that the token's tenant matches the user's existing tenant (users cannot switch tenants via invite).

4. **Last-Login Update**: The user's `last_login` timestamp is updated.

5. **JWT Issuance**: A JWT token is created and returned.

**TUI-Specific Login (`POST /api/tui/auth/login`)**:

The terminal UI has a dedicated password login endpoint that bypasses CSRF protection (since the TUI is not a web browser). This endpoint is added to the public endpoints list so authentication is not required to reach it, but the password is still validated.

### 3.3 JWT Token Structure and Usage

When authentication succeeds, the backend creates a JWT token containing the following claims:

- **`sub` (subject)**: The user's unique ID (e.g., `google_12345` for OAuth users, or `pwd_abc123` for password users).
- **`email`**: The user's email address.
- **`tenant_id`**: The numeric ID of the tenant (used for RLS enforcement).
- **`tenant_subdomain`** (optional): The tenant's subdomain, included for frontend URL redirects.
- **`exp`**: The token expiration time (typically 30 minutes from issuance).

**Token Signing**: Tokens are signed using HS256 (HMAC-SHA256) with the server's `JWT_SECRET_KEY`. This symmetric algorithm means the server uses the same secret to sign and verify tokens.

**Token Transmission**: The client stores the token and includes it in subsequent requests via the `Authorization: Bearer <token>` HTTP header. Alternatively, the token may be stored in a secure HTTP-only cookie.

### 3.4 Request Authentication and Tenant Context

On each incoming request, the authentication middleware performs the following steps:

1. **Token Extraction**: The middleware extracts the JWT from the `Authorization` header (or falls back to a cookie).

2. **Token Validation**: The token is decoded and verified using the server's `JWT_SECRET_KEY`. If the signature is invalid or the token has expired, a 401 Unauthorized response is returned.

3. **User Lookup**: The user ID from the token is used to query the database and retrieve the full User record. This ensures the user still exists and is active (not disabled).

4. **Tenant Context Setup**: The `tenant_id` from the token is stored in `request.state.tenant_id` and also set as a PostgreSQL session variable (`app.tenant_id`). This session variable is used by Row-Level Security rules to filter queries to only data belonging to that tenant.

5. **Request Attachment**: The User and Tenant objects are attached to `request.state.user` and `request.state.tenant` so downstream handlers can access the authenticated user's information.

**Dependency Injection**: FastAPI uses the middleware's `get_current_user()` and `get_current_user_with_tenant()` functions as dependency injection points. Endpoints that require authentication declare these as parameters, and FastAPI automatically calls them to validate the token and retrieve the user before invoking the endpoint handler.

### 3.5 Invite-Only Onboarding

For organizations that want to tightly control user access, the platform supports invite-only tenant joining via the `MULTI_TENANT_ENFORCE_INVITE` configuration flag.

**TenantInvite Record**:

Each invite is represented by a `TenantInvite` database record containing:

- **`token`**: A unique, randomly generated token (the invite link).
- **`tenant_id`**: The tenant the user will join upon signup.
- **`email`**: (Optional) The email address this invite is restricted to. If set, only this email can use the invite.
- **`expires_at`**: The date and time when the invite expires.
- **`used_at`**: Set to the current timestamp when the invite is consumed; `NULL` for unused invites.

**Invite Validation Flow**:

When a user signs up or authenticates via OAuth with an invite token:

1. **Token Lookup**: The backend queries for a `TenantInvite` with the matching token.

2. **Expiration Check**: The invite's `expires_at` is compared to the current time. Expired invites are rejected.

3. **Email Validation** (if applicable): If the invite specifies an email, the user's email must match (case-insensitive).

4. **Single-Use Enforcement**: The invite's `used_at` field must be `NULL`. Once an invite is used, it is marked with the current timestamp and cannot be reused.

5. **Tenant Assignment**: If all checks pass, the new user is assigned to the invite's tenant instead of creating a new tenant.

6. **Prevent Tenant Switching**: If an existing user tries to log in with an invite that belongs to a different tenant, the login is rejected (users cannot switch tenants).

This flow ensures that organizations can control who gains access to their workspace by creating time-limited, email-specific invites that are consumed on first use.

### 3.6 Security Considerations

**Password Hashing**: Password hashes use PBKDF2-SHA256 with 200,000 iterations. This computational cost means an attacker would need significant computing power to brute-force passwords, even if they obtained the hashed password file.

**JWT Security**: The `JWT_SECRET_KEY` must be kept confidential and rotated periodically in production. If compromised, an attacker could forge tokens.

**CSRF Protection**: Password endpoints include CSRF token validation (except for the TUI endpoint, which bypasses CSRF since it is not browser-based).

**Rate Limiting**: Signup and login endpoints are protected by rate-limiting middleware to prevent brute-force and dictionary attacks.

**IP Enforcement** (optional): When enabled, the system can block signup from VPN/proxy IPs or limit one account per IP, reducing fraudulent account creation.

**RLS Enforcement**: PostgreSQL Row-Level Security rules prevent SQL injection or authorization bypasses from exposing data across tenants. Even if an attacker compromises a user session, they can only query data for their tenant.

**Token Expiration**: Short-lived tokens (30 minutes by default) reduce the window in which a stolen token can be used. Refresh tokens with longer expiration (7 days) allow clients to obtain new access tokens without requiring re-authentication.

## 4. API Layer: GraphQL and Middleware

The Etherion AI platform exposes its API through a dual-layer architecture combining GraphQL for application logic with a thin REST layer for specialized operations. This design prioritizes developer experience, real-time capabilities, and secure multi-tenant isolation.

### GraphQL Schema Structure

The GraphQL API is implemented using Strawberry, a Python-native GraphQL library that operates as a mounted router at `/graphql`, accepting both HTTP POST requests and WebSocket connections for subscriptions. The schema is built from three root types: Query, Mutation, and Subscription.

**Queries: Read Operations**

Queries are grouped functionally by domain:

*Authentication and User Context*: `getCurrentUser` returns the authenticated user's profile; `getUserSettings` retrieves user preferences scoped to the tenant.

*Organizational Resources*: `getProjectsByTenant` and `getConversationsByProject` provide hierarchical navigation through projects and their associated conversations; `getToneProfiles` returns AI persona configurations for voice and tone customization.

*Execution and History*: `listThreads` and `listMessages` expose the threading model for multi-turn AI conversations; `getThread` retrieves a specific thread with full message history. `getJobHistory` and `getJobDetails` query past job executions; `getArchivedTraceSummary` provides historical trace data for debugging completed jobs.

*Integration and Tools*: `listRepositoryAssets` enumerates files and artifacts stored in the artifact repository (MinIO backend); `listAgentTeams` and `getAgents` explore configured agent definitions; `getIntegrations` lists connected third-party integrations (OAuth tokens, API keys); `getAvailableMCPTools` surfaces the Model Context Protocol tools available for the tenant.

*Operational*: `getVendorQuotaRemaining` returns daily API quotas per integration vendor (Gmail, Slack, Jira, etc.); `health_check` is a liveness probe.

**Mutations: Write Operations**

Mutations are grouped by resource type and operational intent:

*Project Management*: `createProject`, `updateProject`, and `deleteProject` manage workspace projects; `createConversation` and `renameConversation` manage conversations within a project.

*Agent Configuration*: `createCustomAgentDefinition` defines a custom AI agent with specific prompts and capabilities; `createAgentTeam` groups agents for coordinated execution; `createAgent`, `updateAgent`, and `deleteAgent` manage team membership; `deleteAgentTeam` removes a team; `cloneSystemAgent` creates a custom variant of a built-in agent template.

*AI Execution (Core)*: `executeGoal` is the primary entry point for AI work—it accepts a goal description, optional agent team, and project context, then enqueues a long-running job; `executeAgent` directly invokes a specific agent without orchestration; `cancelJob` terminates a running or queued job.

*Profile and Personalization*: `createToneProfile` defines a new tone/voice profile for AI responses; `updateUserSettings` modifies user-scoped preferences (defaults, features, privacy).

*Integration Lifecycle*: `connectIntegration` initiates OAuth flow or stores API credentials; `testIntegration` validates credentials by making a test API call; `disconnectIntegration` revokes or deletes stored credentials; `executeMCPTool` invokes a Model Context Protocol tool with parameters.

*Tenant and Multi-Tenancy*: `createTenant` provisions a new tenant (typically admin-only); `createTenantInvite` generates an invite token for onboarding team members; `updateTenantSubdomain` allows rebranding.

*Scheduled Execution*: `createScheduledTask`, `updateScheduledTask`, and `deleteScheduledTask` manage recurring AI jobs.

*File Ingestion*: `updateFileIngestionStatus` tracks multimodal ingestion progress.

*Feedback and Support*: `submitFeedback` logs user feedback; `handleSupportTicket` opens support requests; both are audit-logged.

**Subscriptions: Real-Time Updates**

Subscriptions use Redis pub/sub channels to deliver live events to connected WebSocket clients:

*Job Status Monitoring*: `subscribeToJobStatus(job_id)` yields periodic status updates (`QUEUED`, `RUNNING`, `COMPLETED`, `FAILED`) with progress percentage, current step description, and terminal error messages. The subscription maintains isolation—clients can only subscribe to jobs they own (user-level verification).

*Execution Trace Events*: `subscribeToExecutionTrace(job_id)` streams step-by-step trace events as the AI executes: tool invocations, LLM API calls, branching decisions. Each event includes a timestamp, step description, and optional structured output. The API enforces per-IP connection limits (10 concurrent WebSocket connections) and message throttling (60 events per minute per client) to prevent resource exhaustion.

*Tenant-Scoped UI Events*: `subscribeToUIEvents(tenant_id)` delivers generic UI broadcast events (new agent deployed, quota warning, integration connected) to all users in a tenant. Access is tenant-verified.

*Multi-Job Dashboard*: `subscribeToJobUpdates(tenant_id)` streams status updates for all jobs in a tenant, useful for dashboard monitoring.

All subscriptions authenticate via JWT in the WebSocket `connection_params` header, verify tenant ownership, and cleanly terminate on job terminal state or client disconnect.

### Middleware Stack

The middleware pipeline executes in a precise order, with each layer adding security, observability, or request enrichment:

**1. CORS (Starlette CORSMiddleware)**

Configurable per environment via `CORS_ORIGINS` and `CORS_ORIGIN_REGEX` environment variables. Allows credentialed requests (cookies, auth headers) to cross-origin clients, critical for SPAs and TUI integrations. By default permits origins matching `etherionai.com` domains.

**2. Security Integration (Unified Handler)**

A custom middleware layer (`secure_request_handler`) coordinating rate limiting, CSRF protection, security headers, and audit logging. It executes before downstream handlers and runs these checks in order:

- **Rate Limiting**: Per-IP request counting (default 120 req/min, configurable). Uses Redis for state; exempts `/health`, `/oauth`, `/webhook` paths. Returns `429 Too Many Requests` with `Retry-After` header when exceeded.
- **CSRF Protection**: For state-changing methods (POST, PUT, DELETE), validates CSRF token from request header against session and user ID. Exempts GraphQL and webhook endpoints (which use OAuth signatures or Bearer auth as CSRF defense).
- **Auth Check**: Extracts Bearer token from `Authorization` header, decodes JWT, populates `request.state.user_id` and `request.state.tenant_id`.
- **Security Headers**: Post-response, adds `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Strict-Transport-Security` (in production), etc.
- **Audit Logging**: Records authentication successes/failures, authorization denials, rate limit violations, security violations to structured JSON logs.

The security layer fails closed: invalid CSRF tokens trigger 403, missing auth on protected endpoints returns 401, rate limit exceeded returns 429.

**3. Tenant Middleware**

Reads tenant context from JWT claims and sets `request.state.tenant_id`. This value is then propagated to the Postgres session to enable Row-Level Security (RLS) filtering. The tenant ID forms the isolation boundary: no user query can return data outside their assigned tenant.

**4. Auth Context Middleware**

Resolves the current user from JWT and attaches `request.state.auth_context` containing `current_user` (User ORM object), `db_session` (optional scoped session), and resolved `tenant_id`. For WebSocket/GraphQL requests lacking `request.state`, it falls back to parsing headers from the GraphQL context or WebSocket connection params.

**5. Request Logger**

Generates a structured JSON log entry for every request, including request ID, method, path, user ID, tenant ID, client IP, response status, and duration. Logs are emitted to stdout for collection by systemd journal or container log aggregators.

**6. Versioning Middleware**

Reads `API-Version` header (e.g., `v1`, `v2`) and sets `request.state.api_version`. Allows gradual API evolution without breaking clients.

**7. GraphQL Logger**

For GraphQL requests, logs the operation name (query or mutation), parameter count, and execution duration. Separate from the request logger to capture GraphQL-specific metrics.

**8. Error Handler**

Catches exceptions and formats them as structured JSON responses. GraphQL errors are reformatted to include `message`, `extensions`, and optional `path` fields. REST exceptions (HTTPException) are serialized with status code and detail. Unhandled exceptions log a 500 Internal Server Error.

**Order Matters**: CORS runs first to permit cross-origin requests; security checks (rate limit, CSRF, auth) run before reaching handlers so invalid requests are rejected early. Tenant and auth context enrichment must complete before business logic. Error handling wraps the entire chain to catch and format exceptions from any layer.

### REST Endpoints

Alongside GraphQL, a small set of REST endpoints handle specialized operations:

*Health and Liveness*: `GET /health` returns `{"status": "OK"}` for load balancer health checks.

*Authentication*: `POST /auth/login` accepts email and password, returns JWT and user metadata (used by web client); `GET /auth/oauth` initiates browser-based OAuth with a provider (Google, GitHub, Microsoft, Slack); `GET /auth/callback/:provider` handles the OAuth redirect and exchanges code for token.

*TUI-Specific*: `POST /api/tui/auth/login` is identical to `/auth/login` but CSRF-exempt, designed for headless TUI login; `POST /api/tui/oauth/token/:provider` allows the TUI to store personal access tokens (Linear, Notion, Jira) without triggering full OAuth flows.

*Webhooks*: `POST /webhook/admin/ingest-bytes` and `POST /webhook/admin/purge-eval-state` are internal admin endpoints for KB ingestion triggers and state cleanup.

*Asset Access*: `GET /assets/:asset_id` retrieves asset metadata and preview links; `GET /assets/:asset_id/download` generates a time-limited signed URL to download from MinIO.

*Repository Management*: `GET /repo/assets` lists assets for a job; `GET /repo/assets/:asset_id` gets asset details; `POST /repo/assets/:asset_id/access` fetches access credentials.

*OAuth Integration*: `GET /oauth/:vendor/start` initiates OAuth flows for integrations (Gmail, Slack, Jira, Notion, Shopify, etc.); `GET /oauth/:vendor` handles the callback and stores credentials securely in Vault; `GET /oauth/silo/:provider/start`, `GET /oauth/silo/:provider/callback`, `GET /oauth/silo/:provider/status` manage single-sign-on integrations separate from tool integrations.

*Admin Webhooks* (secured by `ADMIN_INGEST_SECRET` header):
  - `POST /webhook/admin/ingest-bytes` ingests raw file content for CDC smoke testing
  - `GET /webhook/admin/ingest-status/:job_id` polls ingestion job status
  - `POST /webhook/admin/purge-eval-state` (with `dry_run` flag) purges tenant data for testing
  - `POST /webhook/admin/repo/create-asset` and `POST /webhook/admin/repo/access-asset` create and access test assets

*Vendor Webhooks* (signature-verified):
  - `POST /webhook/slack/:tenant_id` handles Slack event callbacks (verified via `X-Slack-Request-Timestamp` and `X-Slack-Signature`)
  - `POST /webhook/jira/:tenant_id` handles Jira webhooks (verified via `X-Atlassian-Webhook-Signature`)
  - `POST /webhook/notion/:tenant_id` handles Notion change events (verified via `X-Notion-Signature`)
  - `POST /webhook/shopify/:tenant_id` handles Shopify store events (verified via `X-Shopify-Hmac-Sha256`)

Each vendor webhook verifies the signature, enforces per-vendor daily quotas (e.g., 5000 Slack events/day), and enqueues a Celery task for async processing.

*Operational*: `DELETE /tenant/:tenant_id` enqueues a GDPR deletion task; `POST /secrets/:tool/:key/rotate` rotates stored credentials and revokes old tokens where supported.

All REST endpoints enforce authentication via JWT in the `Authorization` header (except public endpoints like `/health` and `/auth/login`). Tenant isolation is enforced: users can only access resources in their tenant. Errors are returned as JSON with appropriate HTTP status codes (401 Unauthorized, 403 Forbidden, 404 Not Found, 429 Too Many Requests, 500 Internal Server Error).

## 5. Orchestration: IO, Agent Teams, and the 2N+1 Loop

Etherion's orchestration layer sits between user intent and execution. It is built on a dual-orchestrator architecture: **IO** (the platform-level orchestrator) and **TeamOrchestrator** (the team-level orchestrator). This separation of concerns enables the platform to reason about goals holistically while delegating focused execution to specialist teams. The underlying execution model is the **2N+1 loop**: N specialist agents contribute their findings, then a final synthesis step integrates those findings into a coherent response.

### 5.1 What is IO?

**IO** is the name of the top-level AI agent—a LangChain-powered LLM that operates within a single tenant's scope. IO cannot cross tenant boundaries; all operations are enforced by tenant context at the request level.

IO's core responsibilities are:

1. **Answer questions with dual search**: For every user query, IO performs parallel search against both the tenant's Knowledge Base (pgvector) and the public web (via the configured web search tool). This dual grounding eliminates hallucination; if a fact is not found in either source, IO says so explicitly.

2. **Design and propose Agent Teams interactively**: When a goal doesn't fit existing teams, IO guides the user through a blueprint process. This is not automatic team creation—blueprints must be reviewed and approved by the user before instantiation. IO uses registry-first tool validation to prevent hallucinated tool names from entering the blueprint.

3. **Select and delegate to existing teams**: Given a goal, IO queries the tenant's team catalog, evaluates fitness, and recommends the best team. It provides quick-launch links and a summary of expected execution.

4. **Enforce tool approval policy**: IO validates that any tool proposed for use is (a) registered in the ToolManager registry, (b) pre-approved for the selected team, (c) credentialed with tenant secrets, and (d) for writes, confirmed by the user via a confirm-action gate. This is a fail-closed policy: if any condition is unmet, IO rejects or escalates.

5. **Publish execution trace events to Redis**: As IO reasons, it emits structured trace events to Redis channels (namespaced by job ID) and simultaneously logs to stdout. This enables real-time WebSocket subscriptions on the client side, showing users IO's decisions as they happen.

Critically, **IO does not execute tools directly**. It designs plans, checks constraints, recommends actions, and delegates to team runtimes. This separation preserves clarity about who is making decisions and who is executing them.

### 5.2 The User Goal Flow: From Intent to Response

A user goal flows through the system in this order:

1. **User submits a goal** (e.g., "Summarize Q4 performance and propose cost savings"):
   - The goal is routed to IO with tenant context, user identity, and available credits.

2. **IO performs dual search**:
   - Searches the tenant Knowledge Base for relevant documents, metrics, and context.
   - Simultaneously searches the web for industry benchmarks, best practices, and external data.
   - Cites all sources and flags any conflicts.

3. **IO evaluates existing teams**:
   - Queries the database for all active teams in the tenant.
   - For each team, checks whether specialist agents and pre-approved tools fit the goal.
   - Scores teams by relevance, cost, and latency.

4. **IO recommends a team** (or proposes creating one):
   - If a high-confidence match exists, IO recommends it with reasoning.
   - If no match, IO offers to design a bespoke team through the blueprint process (requiring user approval).
   - IO surfaces the expected timeline, cost, and any guardrails (e.g., approval gates).

5. **TeamOrchestrator executes the 2N+1 loop**:
   - The selected team's orchestrator is instantiated with the goal.
   - N specialist agents are invoked in parallel, each addressing a facet of the goal.
   - A final synthesis step aggregates specialist outputs.

6. **Result flows back through IO**:
   - IO reviews the synthesis, validates it against the original goal, and delivers it to the user.
   - If any tool failed or context is incomplete, IO escalates or offers next steps.

### 5.3 The 2N+1 Loop: Why This Structure?

The 2N+1 loop is a reasoning architecture that improves output quality over single-agent chains.

**Structure**: Given a goal, the loop invokes N specialist agents in parallel, waits for all to complete, then runs one final synthesis step.

**Example**: For the goal "Analyze sales pipeline and forecast Q1 revenue," a team might define:

- **N specialists** (run in parallel):
  1. Sales Data Analyst: Extracts pipeline metrics, conversion rates, deal velocity.
  2. Forecast Specialist: Applies statistical models and historical trends.
  3. Risk Analyst: Identifies outliers, dependencies, and failure modes.

- **Synthesis step** (runs after all N specialists):
  - Integrates all three analyses into a unified forecast with confidence intervals and risk flags.

**Why it works**:

- **Diversity of perspective**: Each specialist is narrowly focused (e.g., forecast specialist uses only statistics tools). This prevents one agent from dominating reasoning.
- **Parallelism**: Specialists run concurrently, reducing latency. A single sequential chain would take 3x longer.
- **Validation loop**: The synthesis step can detect contradictions (e.g., specialist A says revenue grows, specialist B flags risks). Conflicts are surfaced, not hidden.
- **Tool containment**: Each specialist has a pre-approved tool list. A forecast specialist cannot accidentally call a data-deletion tool.
- **Observability**: Each specialist's reasoning, tool calls, and outputs are emitted as trace events. Users see exactly how the answer was constructed.

By contrast, a single agent given all tools and all data might make arbitrary tool choices, miss nuances, and produce opaque outputs. The 2N+1 structure trades raw capability for clarity and robustness.

### 5.4 Agent Teams: Database-Driven, Not Hardcoded

Agent teams are defined in the database, not baked into code. This enables both static team definitions (created by admins) and dynamic ones (created by users via IO's blueprint process).

**Team Structure** (stored in the `AgentTeam` table):

- **agent_team_id**: A unique identifier (e.g., `at_abc123xyz`).
- **name & description**: Human-readable metadata (e.g., "Financial Analysis Team").
- **custom_agent_ids**: A JSON list of CustomAgentDefinition IDs that are members of this team.
- **pre_approved_tool_names**: A JSON list of tool names that the team is allowed to call.
- **is_system_agent**: If true, this is a platform-owned team that can be cloned by tenants and customized.
- **version, is_latest_version**: Teams are versioned; users can roll back or fork a version.

**Specialist Definition** (stored in the `CustomAgentDefinition` table):

- **custom_agent_id**: Unique identifier (e.g., `ca_analyst_xyz`).
- **system_prompt**: The LLM system prompt that defines behavior and expertise (e.g., "You are a data analyst specializing in...").
- **tool_names**: JSON list of tools this specialist can invoke.
- **model_name**: Which LLM to use (e.g., `gemini-2.5-flash`).
- **temperature, max_iterations, timeout_seconds**: Runtime configuration for consistency.

**Loading at Runtime** (src/services/agent_loader.py):

1. When TeamOrchestrator executes, it calls `AgentLoader.load_agent_team()`.
2. AgentLoader fetches the AgentTeam row from the database.
3. For each specialist ID in the team, it fetches the CustomAgentDefinition.
4. It also loads the UserObservation context (learned preferences) for the requesting user.
5. All definitions are cached in memory for the duration of the execution; no database round trips during specialist loops.

This design means:

- **Admins** can create base teams via SQL or GraphQL and publish them as system teams.
- **Users** can clone system teams and customize them (modify specialists, add/remove tools).
- **IO** can create new teams on the fly by calling `AgentTeamCreator.create_blueprint()`, which invokes an LLM to design the blueprint, validates tool names against the registry, and persists the team to the database.
- **Teams are discoverable**: GraphQL queries can list all teams for a tenant, filter by tags, and return execution history.

### 5.5 The Tool Approval System: Fail-Closed Policy

Tools are the hands of the orchestration system. Every tool call is guarded by a strict approval policy.

**Tool Registry** (source of truth, in code):

- Centralized in `src/tools/tool_registry_tool.py`, which reads from the `ToolManager` class.
- Contains the canonical list of deployed tools (e.g., `unified_research_tool`, `multimodal_kb_search`, `fetch_document_content`).
- No hallucinated tools can exist; only tools in this registry are valid.

**Approval Flow**:

At execution time, when a specialist or IO attempts to call a tool, the orchestrator checks:

1. **Is the tool registered?** Query the ToolManager registry. If not found, reject.
2. **Is it pre-approved for this team?** Check the team's `pre_approved_tool_names` list. If not in the list, reject.
3. **Are tenant credentials available?** Look up any OAuth tokens or API keys required. If missing, reject and offer to connect the credential.
4. **For write operations, is user confirmation recorded?** Check the user's session for a confirmed action. If not, surface a confirm-action prompt and wait for approval.

**Why fail-closed?**

- **Security**: A misconfigured team cannot accidentally gain access to a tool. Defaults are "no access."
- **Cost control**: Tools with high per-call costs (e.g., external API calls) can be intentionally restricted to certain teams.
- **Auditability**: Every tool call is logged with the team, specialist, and user who triggered it.
- **Determinism**: If a blueprint LLM hallucination included a fake tool, blueprint creation fails immediately (registry validation step). The invalid blueprint is never persisted.

**Blueprint Creation with Registry Validation** (src/services/platform_orchestrator.py):

When IO creates a blueprint, it:

1. Calls `tool_registry_tool(include_beta=True, include_deprecated=False, limit=500)` to fetch all valid tool names.
2. Injects this list into the LLM prompt as a hard allowlist.
3. Instructs the LLM: "Choose only from this list; do not invent tools."
4. Validates the returned `tool_requirements` against the allowlist.
5. If any tool name is not in the registry, raises an error and rejects the blueprint. The user must revise.

This prevents the creation of teams with hallucinatory tool capabilities, ensuring E2E reproducibility in evaluation and production.

### 5.6 Execution Trace and Observability

As orchestration executes, every significant decision and action is emitted as a trace event.

**What gets traced**:

- **DUAL_SEARCH**: IO performed a Knowledge Base + Web search; includes result counts and citations.
- **SPECIALIST_REQUEST**: A specialist agent is about to be invoked; includes agent ID, instruction.
- **SPECIALIST_RESPONSE**: Specialist completed; includes duration, success/failure, output snippet.
- **TOOL_START**: A tool is being invoked; includes tool name, input preview.
- **TOOL_END**: Tool completed; includes duration, success/failure, output preview.
- **FILE_FETCH_START/END**: When KB documents are fetched (multimodal paradigm), includes MinIO URI and filename.
- **SYNTHESIS_START/END**: The final synthesis step is running; includes input count and output.

**Where they go**:

- **Redis Pub/Sub** (channel `job_trace_{job_id}`): For live WebSocket subscriptions. Clients subscribe to this channel and receive trace events in real time, enabling live progress UI.
- **stdout**: Same events logged in structured format (`[TRACE] ts=... job_id=... type=... | ...`) for log collection and observability tooling.
- **Archive** (database, via `getArchivedTraceSummary` GraphQL query): After job completion, a summarized trace is stored for later retrieval and audit.

This layered tracing ensures that:

- **Transparency**: Users see what IO and specialists are doing in real time.
- **Debuggability**: If a job fails, the full trace is available for root-cause analysis.
- **Compliance**: Execution traces can be retained for audit and regulatory purposes.
- **Cost visibility**: Trace events include token counts and model latencies, enabling cost per-step visibility.

### 5.7 Conclusion: Orchestration as a Trust Layer

The dual-orchestrator architecture—IO at the platform level, TeamOrchestrator at the team level, grounded in the 2N+1 loop—serves as a trust layer. It ensures that:

- Goals are understood holistically before execution (IO's reasoning).
- Specialists are narrowly scoped and containable (pre-approved tools, system prompts).
- Tool use is guarded by policy, not hope (fail-closed approval).
- Decisions are transparent and auditable (trace events).
- Errors can be caught early (blueprint validation, synthesis checks).

This is why Etherion can safely orchestrate complex AI workflows at scale: the architecture makes misbehavior observable and recoverable before it happens.

## 6. Asynchronous Jobs and the Worker

### Why AI Orchestration Must Be Asynchronous

AI orchestration runs are inherently long-running operations that can take anywhere from seconds to minutes to complete. During this time, the orchestrator executes multiple sequential steps: it may call LLM APIs to reason about a goal, dispatch specialized agents to gather information, invoke external tools such as search or data retrieval services, and iteratively refine results. Each of these operations involves network I/O and can introduce latency.

In a synchronous HTTP architecture, an API server thread would block while waiting for the orchestrator to finish. This creates two critical problems. First, a typical application server (FastAPI, Django, Flask) has a limited thread pool—often just a few dozen threads across all CPU cores. If each long-running orchestration consumes one thread for minutes, the thread pool becomes exhausted and subsequent requests are queued or rejected, causing cascading failures. Second, HTTP has finite timeout windows (often 30 seconds to a few minutes). Long orchestrations would inevitably timeout before completion, forcing the caller to either retry blindly or abandon the work entirely.

Asynchronous job processing solves this by decoupling request handling from execution. When a user calls `executeGoal`, the API immediately returns a `job_id` to the client and queues the actual work to a separate worker process. The client then subscribes to real-time updates via GraphQL subscriptions, and the worker processes the job at its own pace without blocking the API. This design allows the API to handle thousands of concurrent user requests while a small fleet of workers—each configured with appropriate concurrency and resource limits—executes the actual orchestrations.

### Job Lifecycle: From Mutation to Completion

The flow of a single job from inception to completion follows these steps:

1. **Mutation: `executeGoal` called.** The client sends a GraphQL `executeGoal` mutation with a goal description, optional context, and execution hints (provider, model, search preferences, etc.). The API receives the request and authenticates the user.

2. **Job record created.** Inside the mutation resolver, a new `Job` row is inserted into PostgreSQL with `status=QUEUED`, capturing the tenant ID, user ID, goal text, and metadata (provider/model preferences, agent team ID, thread ID). The `job_id` is generated as a URL-safe random identifier (e.g., `job_abc123xyz`). If no thread ID is provided, a new thread row is created to contextualize this job within a conversation. The job's input data (sanitized goal, context, output format instructions) is serialized as JSON and stored in the database.

3. **Job ID returned immediately.** The mutation responds to the client with `JobResponse(success=true, job_id=..., status=QUEUED)`. The client now has a handle to track the work.

4. **Celery task enqueued.** The API calls `execute_goal_celery_task.apply_async()` to dispatch a Celery task with the job's parameters (job_id, goal text, user_id, tenant_id). The task is routed to the `worker-agents` queue and placed in the broker (Redis). Celery's broker stores the task with a unique task ID (which may differ from the job_id) until a worker picks it up.

5. **Client subscribes to updates.** Meanwhile, the client opens two WebSocket subscriptions: `subscribeToJobStatus(job_id)` and `subscribeToExecutionTrace(job_id)`. Both subscriptions are secured by authentication and authorization checks to ensure the user can only see their own jobs. The GraphQL subscriptions connect to Redis Pub/Sub channels and begin listening for updates:
   - `job_status_{job_id}`: Carries high-level status transitions (QUEUED → RUNNING → COMPLETED/FAILED/CANCELLED) with progress percentage, current step description, and any error messages.
   - `job_trace_{job_id}`: Carries fine-grained execution trace events that describe exactly what the AI did—which agent/specialist ran, which tool was invoked, what data was retrieved, and what decision was made.

6. **Worker receives and begins execution.** A Celery worker thread (or process, depending on configuration) picks the task from the broker. It calls `orchestrate_goal_task()` with the job parameters. The orchestrator now transitions the job status to RUNNING in both the database and publishes a status update to Redis (visible to subscribed clients).

7. **Orchestrator runs, publishing trace and status.** As the orchestrator executes, it:
   - Calls LLMs to reason about the goal (recording token usage and cost).
   - Dispatches agents to specialized tasks (research, tool invocation, data processing).
   - Records execution trace events after each step (e.g., `{"type": "agent_selected", "agent_name": "ResearchAgent", ...}`).
   - Publishes these events to the `job_trace_{job_id}` channel so the client sees real-time progress.
   - Updates `job_status_{job_id}` with progress percentages and current step descriptions.
   - Records cost data in Redis (tokens consumed, API calls made, data transferred) for later billing.

8. **Job reaches terminal state.** Once the orchestrator finishes (either successfully or with an error), it:
   - Updates the job status to COMPLETED or FAILED in the database.
   - Serializes the final output (or error details) into the `output_data` JSON field.
   - Publishes a final status event to `job_status_{job_id}` with status=COMPLETED or FAILED.
   - Publishes a closing trace event (e.g., `{"type": "orchestration_complete", ...}`).
   - The subscriptions receive these terminal events and close.

9. **Client receives final result.** The client's subscription handlers process the terminal events and display the result to the user. The `job_id` can then be used to query historical job data, logs, and cost details.

### Worker Configuration and Rationale

The Celery worker for Etherion AI is configured via command-line arguments and environment variables:

```bash
celery worker \
  --loglevel=info \
  --concurrency=8 \
  --max-tasks-per-child=100 \
  --prefetch-multiplier=1 \
  --without-gossip \
  --without-mingle \
  --without-heartbeat \
  --pool=threads \
  --queues=worker-agents
```

Each setting addresses a specific operational concern:

- **`--concurrency=8`** (configurable via `CELERY_CONCURRENCY` env var): The number of concurrent execution slots. With 8 threads, the worker can handle 8 orchestrations in parallel. This number should be tuned based on available system memory and CPU cores. Too high causes memory bloat and context-switching overhead; too low wastes resources. For typical cloud deployments, 8–16 is standard.

- **`--pool=threads`** (not processes): Uses a thread pool instead of a process pool. This is essential because the orchestrator tasks use Python's `asyncio` for concurrent I/O (LLM API calls, external tool invocations). Process-based workers (fork pool) conflict with asyncio; threads allow asyncio event loops to manage I/O cooperatively within each worker slot.

- **`--prefetch-multiplier=1`**: Each worker pre-fetches only 1 task from the broker at a time (instead of the default 4). This prevents one worker from hoarding many tasks while others sit idle. With `prefetch=1`, tasks are distributed fairly across multiple workers, and if a worker crashes, only 1 task is lost (not 4). This is crucial for reliability in production.

- **`--max-tasks-per-child=100`**: After completing 100 tasks, the worker thread exits and a new one is spawned. This prevents memory leaks from long-lived Python objects that might accumulate over time (e.g., internal caches, LLM SDK state). A lower value ensures fresher processes but increases overhead; 100 is a reasonable balance.

- **`--without-gossip --without-mingle --without-heartbeat`**: Disables inter-worker gossip (cluster state synchronization), mingle (initial worker synchronization), and heartbeat pings. These features add network overhead and are unnecessary in a bare-metal or containerized environment where workers are controlled via orchestration (Systemd, Kubernetes). Disabling them speeds up startup and reduces chattiness.

- **`--queues=worker-agents`**: The worker subscribes to the `worker-agents` queue. Some tasks (heavy ingestion, maintenance) are routed to `worker-artifacts` queue and handled by separate worker instances, preventing data-heavy tasks from blocking orchestration work.

### Broker and Result Backend

The worker's broker and result backend are both Redis:

- **Broker** (`CELERY_BROKER_URL`): Tasks are published to a Redis list and workers consume from it. Redis is chosen for low latency, durability, and built-in Pub/Sub support for status updates.

- **Result Backend**: Celery's result backend (where task results are stored) is **disabled by default in production** (`CELERY_RESULT_BACKEND` not set or set to `disabled://`). This is because Etherion AI does not rely on Celery's built-in task result storage. Instead, job results are persisted directly to the PostgreSQL `Job` table and status/trace updates are published via Redis Pub/Sub. This design avoids redundancy and simplifies the architecture. The result backend can be re-enabled for testing or if a future use case requires Celery AsyncResult queries.

### Cancellation: Cooperative Stop Signals

Job cancellation is implemented as a cooperative signal rather than a hard kill:

1. **`cancelJob` mutation called.** The client invokes the `cancelJob(job_id)` mutation to request job termination.

2. **Cancel flag set in Redis.** The mutation calls `set_job_cancel(job_id)`, which sets a boolean flag in Redis at key `job_cancel_{job_id}` with a 1-hour TTL. This flag is a soft signal.

3. **STOP_INTENT event published.** The mutation publishes a trace event `{"type": "STOP_INTENT", "step_description": "User requested to stop the job"}` to alert all subscribers that cancellation was requested.

4. **Best-effort Celery revoke attempted.** If the job corresponds to a long-running Celery worker task (which may not be the case if execution is already complete), the mutation calls `safe_revoke_job(job_id, terminate=True)` to signal Celery. This is a best-effort operation and is safe even if the task ID is unknown or not running.

5. **Job status set to CANCELLED.** The mutation immediately updates the job status to CANCELLED in PostgreSQL and publishes a `job_status_{job_id}` event with status=CANCELLED.

6. **Running orchestrator checks the flag.** Meanwhile, if the orchestrator is still running, it periodically checks `is_job_cancelled(job_id)` (e.g., between orchestration steps or at the start of each agent execution). When it detects the flag is set, it:
   - Stops scheduling new work.
   - Cleans up in-progress operations gracefully (waits for current LLM call to finish, etc.).
   - Publishes a trace event confirming cancellation (e.g., `{"type": "orchestration_cancelled", ...}`).
   - Publishes a final status update with status=CANCELLED and appropriate message.
   - Exits cleanly, allowing the Celery task to complete.

This design ensures that cancellation is always graceful—never a hard kill that could leave resources in an inconsistent state or leave half-processed data dangling. Even if the orchestrator does not check the flag frequently, the job is immediately marked CANCELLED in the database so the UI reflects the user's intent, and any actual cleanup happens asynchronously.

### Cost Tracking: Usage Recording and Attribution

As the orchestrator executes, it records detailed cost data in Redis for later reconciliation and billing:

**Recording and Aggregation:**

The `CostTracker` service (in `src/services/pricing/cost_tracker.py`) exposes methods to record various usage events. All counters are stored in Redis with a key pattern `cost:{tenant_id}:{job_id}:{metric}` for fine-grained attribution:

- **Token usage:** `record_tokens(job_id, input_tokens, output_tokens, tenant_id)` increments counters for input and output tokens per LLM call.
- **API calls:** `record_api_call(job_id, provider, tenant_id)` tracks how many times each LLM provider (Gemini, Claude, OpenAI) was called.
- **Data transfer:** `record_data_transfer(job_id, mb_in, mb_out, tenant_id)` records megabytes of data ingested and output.
- **Compute time:** `record_compute_time_ms(job_id, ms, tenant_id)` logs milliseconds spent in compute-intensive operations.
- **LLM context:** `set_llm_context(job_id, provider, model, mode, tenant_id)` attaches the specific LLM model and mode (e.g., "reasoning" for extended thinking) to the job for accurate pricing lookup.
- **Tool-specific usage:** Methods like `record_exa_search()` and `record_exa_contents()` track usage of external third-party APIs (e.g., Exa web search).

All of these counters are stored in Redis with automatic expiration (e.g., 24 hours) to keep the store bounded. The counters use integer keys (Redis INCR operations) to avoid floating-point precision issues.

**Reconciliation and Persistence:**

At the end of each day (or on demand), a scheduled Celery task `run_reconciliation` (routed to the `worker-artifacts` queue) reads all accumulated cost counters from Redis and persists them to the PostgreSQL `ExecutionCost` table. This table has fields like:

- `job_id`: The job this cost belongs to.
- `tenant_id`: The owning tenant (for multi-tenant cost reporting).
- `input_tokens`, `output_tokens`: Total tokens consumed.
- `api_call_count`: Number of API invocations.
- `compute_ms`: Total compute time.
- `vendor_costs`: JSON object with per-vendor costs (e.g., `{"gemini": 0.015, "exa_search": 0.05}`).
- `total_cost`: Aggregate cost in USD.
- `recorded_at`: Timestamp of reconciliation.

**Why Redis Counters First, Then Persist:**

Using Redis as a temporary cost buffer (rather than writing directly to PostgreSQL on every token) has multiple advantages:
  - **Performance:** Redis increments are O(1) and extremely fast; the orchestrator is not blocked by database writes.
  - **Atomicity:** A single orchestration job's counters can be atomically retrieved and written to the database in one reconciliation task.
  - **Debugging:** Cost data in Redis can be inspected in real time for monitoring and troubleshooting.
  - **Bounded storage:** With TTLs, Redis doesn't accumulate unbounded cost data.

**Scope and Attribution:**

Cost data is scoped per job and per tenant. This ensures that:
  - Each job's costs are independently tracked and can be attributed to the user who initiated it.
  - Tenants can view their cumulative costs for billing and budget management.
  - The system can enforce per-tenant cost limits (e.g., daily spend caps) by consulting the PostgreSQL `Tenant` table during execution.

Cost tracking runs asynchronously and does not block job execution. If a cost tracking call fails (e.g., Redis connection issue), the job continues—cost tracking is advisory and does not prevent the orchestration from completing. This ensures maximum availability and resilience.

## 7. Knowledge Base and Vector Search

### The Knowledge Base: Tenant-Isolated Document Storage

At its core, the Etherion Knowledge Base (KB) is a per-tenant repository of documents and media that grounds the platform in customer-specific truth. Each tenant builds and maintains a Knowledge Base scoped to projects—allowing multi-tenancy with strict data isolation. The KB stores structured document chunks, their semantic embeddings, and associated metadata in PostgreSQL with pgvector indexing.

The Knowledge Base exists to answer a fundamental question: "What does this tenant actually know?" Instead of relying on externally trained models or general web knowledge, the platform prioritizes grounding all answers in the tenant's own documents first. This principle—called the "Dual Search Principle" (discussed below)—ensures that answers are trustworthy, citable, and contextually accurate to the organization's specific domain.

### Document Hierarchy and Organization

Documents are organized into a logical hierarchy:

- **Projects**: Tenants create multiple projects, each with its own scoped Knowledge Base. A project might represent a department, initiative, or domain.
- **Knowledge Bases**: Each project has one or more Knowledge Bases (via the `KnowledgeBase` SQLModel table), with a `kb_type` field distinguishing between project-scoped and personal Knowledge Bases.
- **Document Chunks**: Raw files are ingested and split into searchable chunks. Each chunk is stored as a row in the `document` table with its embedding, text content, metadata, and a reference to its parent Knowledge Base. Metadata includes source file information, retention policy dates, and processing status.

The multi-level scoping allows fine-grained access control: an agent team can search only within their project's KB, enforcing tenant isolation at query time via the `WHERE tenant_id = :tid AND kb.project_id = :project_id` predicate.

### The Ingestion Pipeline: From File to Searchable Chunk

When a tenant uploads a document (PDF, text, image, or other formats), the system launches an asynchronous Celery task that orchestrates a multi-stage ingestion pipeline:

**Stage 1: Upload and Parse**
The file bytes are first uploaded to MinIO (self-hosted object storage) and stored with a content hash as the path. Concurrently, the `MultimodalIngestionService` parses the document using deterministic parsing:

- For PDFs: PyMuPDF extracts text, structure, and images page-by-page.
- For text files: Raw UTF-8 decoding with token estimation.
- For other formats: Docling or fallback parsers handle Office documents and other media.

The parser produces a `DocumentParseResult` containing markdown-rendered text, extracted chapters (with headings), and embedded images.

**Stage 2: Chunking and Tokenization**
Documents larger than 300,000 tokens are automatically split into parts. Each part maintains chapter boundaries to preserve semantic context. Token counts are estimated using tiktoken's `cl100k_base` encoding, ensuring that long documents are manageable for embedding APIs without losing coherence.

**Stage 3: Embedding**
For each chunk (or part), the system extracts a chapter essence: the heading plus the first 128 tokens of content. This essence is embedded using the configured embedding model. By default, the `EmbeddingService` calls the configured embedding provider to generate vectors (dimension controlled by `KB_EMBEDDING_DIM`, default 1536). The embeddings are computed once at ingestion time and cached in PostgreSQL.

**Stage 4: Storage in PostgreSQL with pgvector**
Each chunk is inserted into the `document` table as a row containing:
- `doc_id`: Unique identifier
- `text_chunk`: The full text of the chunk
- `embedding`: A pgvector column storing the vector (dimensionality configurable via `KB_EMBEDDING_DIM`, defaulting to 1536)
- `metadata_json`: JSON-serialized metadata (filename, page range, chapter info, retention policy)
- `storage_uri`: MinIO path to the original file
- `tenant_id`, `kb_id`: For multi-tenancy and scoping

The pgvector extension allows efficient similarity search using the cosine distance operator.

**Stage 5: Multimodal Enrichment**
Extracted images are uploaded to MinIO and embedded using the multimodal embedding service, placing them in the same vector space as text chunks. This enables cross-modal search: a text query can retrieve relevant images, and vice versa.

### Vector Search: Mechanics and Scoring

The vector search flow follows these steps:

1. **Query Embedding**: When a user or agent poses a search query, the query text is embedded using the same embedding model used at ingestion. This ensures the query vector and document vectors occupy the same semantic space.

2. **Similarity Computation**: The `PgvectorKBBackend` executes a SQL query using pgvector's cosine distance operator:

```sql
SELECT d.doc_id, d.text_chunk, d.metadata_json, d.storage_uri,
       1 - (d.embedding <=> CAST(:vec AS vector)) AS score
FROM document d
JOIN knowledgebase kb ON kb.id = d.kb_id
WHERE d.tenant_id = :tid
  AND (kb.project_id = :project_id OR :project_id IS NULL)
ORDER BY score DESC
LIMIT :k
```

The `<=>` operator computes the cosine distance (a value between 0 and 2). By subtracting from 1, we convert it to a similarity score: `score = 1 - distance`. Higher scores (closer to 1) indicate higher relevance.

3. **Result Ranking**: Results are ordered by score in descending order, returning the top-k most relevant chunks. Each result includes the chunk text, metadata, storage URI, and similarity score for citation and further retrieval.

4. **Tenant and Project Scoping**: Queries are always scoped to the requesting tenant (`tenant_id`) and optionally filtered by project. This prevents cross-tenant data leakage and enforces project-level access control.

The similarity score should be interpreted as a confidence measure: scores above 0.8 typically indicate high relevance, while scores below 0.5 may signal weak matches that warrant hybrid search or web fallback.

### The Dual Search Principle: Knowledge Base + Web

A cornerstone of Etherion's answer quality is the **Dual Search Principle**: for every question or uncertain query, the platform always performs two searches in parallel:

1. **Knowledge Base Search**: Retrieves the tenant's own documents using vector similarity, grounding the answer in customer-specific truth.
2. **Web Search**: Retrieves current external information via search APIs, ensuring freshness and coverage of topics outside the KB.

When results conflict, the Knowledge Base takes precedence—reflecting the principle that "what you know" (your documents) should outweigh general knowledge. However, both sources are cited, allowing the user to understand the evidence trail and make informed decisions.

This dual approach is encoded in the `PlatformOrchestrator` (IO), which follows the mandate: "Perform dual search; synthesize an answer with citations. Prefer KB over Web when conflicts arise; still cite both." The orchestrator emits trace events indicating which sources were consulted, enabling users and auditors to verify answer provenance.

The practical benefit: Etherion avoids hallucination by requiring evidence from at least one of two concrete sources. If both are silent, IO explicitly states the limitation and proposes how to fill the gap (upload documents, allow broader search, etc.).

### Multimodal Support: Text and Images in Unified Vector Space

The Etherion KB extends beyond text to support multimodal search:

- **Image Extraction**: During ingestion, PyMuPDF extracts embedded images (charts, diagrams, photos) from PDFs. Each image is deduplicated by content hash to avoid redundant storage.
- **Image Embedding**: Extracted images are embedded using a multimodal embedding model that produces vectors in the same space as text chunks.
- **Cross-Modal Search**: Because images and text are in the same vector space, a text query like "revenue chart" can retrieve both documents discussing revenue and the actual chart images from those documents. Conversely, uploading a chart image as a query can find similar charts and documents about those topics.
- **Metadata Preservation**: Image embeddings retain metadata including the parent document ID, page number, and chapter context, enabling users to navigate from image back to source.

Multimodal search is particularly powerful for organizations working with reports, presentations, and data visualizations, where domain-specific diagrams and charts carry as much semantic weight as text.

### Retention and Data Lifecycle

Each document chunk is associated with a retention policy:

- Documents uploaded to a Knowledge Base inherit the tenant's default retention policy (e.g., 365 days).
- The `ProjectKBFile` table tracks `file_name`, `file_uri`, ingestion `status` (processing, available, failed), and `retention_policy_days`.
- An `archive_after` field is computed at ingestion and stored, enabling automated cleanup. Once the archive date passes, chunks are eligible for deletion by background jobs.

This lifecycle management ensures compliance with data retention requirements and prevents indefinite storage bloat.

### Backend Abstraction and Extensibility

The vector backend is abstracted behind a `KBBackend` interface in `src/services/kb_backend.py`. The default and recommended implementation, `PgvectorKBBackend`, uses PostgreSQL with the pgvector extension — keeping all vector computation local with no external dependencies. The backend is selected via the `KB_VECTOR_BACKEND` environment variable (default: `pgvector`).

### Search-to-Answer Workflow

A typical user query follows this flow:

1. User asks a question via the Etherion UI.
2. IO (the Platform Orchestrator) receives the query and initiates dual search.
3. In parallel: KB search embeds the query and retrieves top-k relevant chunks; web search queries an external search API for fresh context.
4. IO synthesizes a grounded answer, citing both sources (KB doc IDs and web URLs).
5. If the user wants to dive deeper, they can fetch full document content using the `fetch_document_content` tool, which retrieves and parses the original file from MinIO.
6. Results are presented with confidence levels, caveats, and next-step recommendations (e.g., "To improve this answer, upload the Q4 financial report").

---

The Knowledge Base transforms Etherion from a generic question-answering system into a domain-specific, trustworthy advisor tailored to each tenant's unique information landscape. By anchoring all reasoning in vector-grounded search and maintaining strict tenant isolation, the platform delivers answers users can rely on and audit.

## 8. Real-Time Updates: Redis Pub/Sub and GraphQL Subscriptions

### 8.1 Why Real-Time Updates Need a Pub/Sub System

HTTP requests follow a request-response pattern where the client initiates communication and the server provides an answer before closing the connection. This model does not suit long-running AI jobs that must deliver live feedback to a waiting user.

Traditional polling—where the client repeatedly fetches the latest job status—introduces unnecessary latency, wastes bandwidth, and scales poorly as user counts grow. A client polling every 500 milliseconds on 100 concurrent jobs creates 200 requests per second, straining both the API and the client's event loop.

The platform instead uses a **publish-subscribe (Pub/Sub) pattern** backed by Redis:

- The **backend** (worker, orchestrator) publishes discrete events into Redis channels as they occur.
- The **frontend** (browser or terminal UI) subscribes to those channels via a persistent WebSocket connection.
- Events flow to clients in real time with minimal latency and no request overhead.

This design ensures that users see job progress, error messages, and completed artifacts as soon as they happen, improving perceived responsiveness and reducing the need for manual refresh cycles. The Pub/Sub broker (Redis) naturally fans out messages to all subscribers, making it trivial to support multiple clients watching the same job.

---

### 8.2 The Three Redis Channels

The platform publishes events on three types of Redis channels:

#### 8.2.1 Job Status Channel: `job_status_{job_id}`

**Purpose:** High-level state transitions and progress updates for a specific job.

**When published:**
- When a job transitions between states: `QUEUED` → `RUNNING` → `COMPLETED`, `FAILED`, or `CANCELLED`.
- When progress is updated (e.g., "50% complete").
- When the job completes with a summary message.

**Publisher:** `src/services/job_status_publisher.py` (called by the orchestrator at state boundaries) and `src/core/redis.py::publish_job_status()` helper.

**Typical payload:**
```json
{
  "job_id": "job_abc123xyz",
  "status": "RUNNING",
  "timestamp": "2026-03-26T14:30:00Z",
  "message": "AI synthesis in progress",
  "progress_percentage": 45,
  "current_step_description": "Evaluating search results",
  "error_message": null
}
```

**Consumer:** GraphQL subscription `subscribeToJobStatus(job_id)` in the API. Terminal UI or web app polls this subscription.

---

#### 8.2.2 Job Execution Trace Channel: `job_trace_{job_id}`

**Purpose:** Fine-grained execution events that reveal which specialists, tools, and steps ran during job execution.

**When published:**
- When a specialist is invoked (e.g., "Research Agent starting").
- When a tool is called (e.g., "multimodal_kb_search with query: 'X'").
- When a tool completes with results or an error.
- When a file is fetched from knowledge base storage.
- When a synthesis phase begins or concludes.
- When cost updates are calculated (emitted as `type: "cost_update"`).

**Publisher:** `src/services/team_orchestrator.py`, `src/services/platform_orchestrator.py`, and `src/core/redis.py::publish_execution_trace()` helper. Also `CostTracker.publish_cost_event()`.

**Typical payload (multi-part trace):**
```json
{
  "job_id": "job_abc123xyz",
  "type": "SPECIALIST_REQUEST",
  "specialist_id": "research_specialist_v2",
  "instruction_preview": "Find the top 5 recent publications on ...",
  "timestamp": "2026-03-26T14:30:01Z"
}
```

followed by:

```json
{
  "job_id": "job_abc123xyz",
  "type": "TOOL_START",
  "tool_name": "unified_research_tool",
  "input_preview": "query='machine learning benchmarks'",
  "timestamp": "2026-03-26T14:30:02Z"
}
```

and later:

```json
{
  "job_id": "job_abc123xyz",
  "type": "DUAL_SEARCH",
  "step_description": "KB + Web search executed",
  "counts": {
    "kb_results": 4,
    "web_results": 5
  },
  "timestamp": "2026-03-26T14:30:05Z"
}
```

**Consumer:** GraphQL subscription `subscribeToExecutionTrace(job_id)`. The evaluation harness ingests this as JSONL for analysis. The terminal UI displays a structured timeline. The web UI renders a collapsible execution tree.

---

#### 8.2.3 Tenant UI Events Channel: `ui_events_{tenant_id}`

**Purpose:** Tenant-scoped notifications not tied to a specific job.

**When published:**
- A new team has been created or deployed.
- A knowledge base ingestion batch completed (e.g., "50 documents indexed").
- A configuration change took effect.
- A billing alert is triggered.
- Any other platform-level event relevant to the tenant's operations.

**Publisher:** `src/core/redis.py::publish_ui_event(tenant_id, event_data)`.

**Typical payload:**
```json
{
  "tenant_id": 42,
  "type": "team_created",
  "team_name": "Research Team v3",
  "timestamp": "2026-03-26T14:35:00Z",
  "message": "Team 'Research Team v3' is now ready to use."
}
```

**Consumer:** GraphQL subscription `subscribeToUIEvents(tenant_id)`. The web app dashboard listens to this and displays a toast notification or updates a list.

### 8.3 End-to-End Flow: From Worker Publishing to Client Receiving

The typical flow for a job execution with live updates looks like this:

#### Step 1: Job Starts on Worker

The Celery worker (or equivalent job runner) begins executing the job. The worker immediately publishes:

```python
await publish_job_status(job_id, {
    "status": "RUNNING",
    "message": "Job started",
    "progress_percentage": 0
})
```

This write lands in Redis channel `job_status_{job_id}` with a JSON-serialized payload.

#### Step 2: GraphQL Subscription Resolver Connects

Simultaneously, a user opens the job details page in their browser or terminal UI and subscribes via GraphQL:

```graphql
subscription {
  subscribeToJobStatus(job_id: "job_abc123xyz") {
    job_id
    status
    timestamp
    message
    progress_percentage
  }
}
```

The Apollo client (frontend) opens a WebSocket connection to the API at `wss://api.etherionai.com/graphql` and sends a `connection_init` message with auth headers. Strawberry's GraphQL router receives this on the `/graphql` endpoint and calls the subscription resolver.

#### Step 3: Resolver Validates Auth and Subscribes to Redis

Inside `subscribeToJobStatus` in `src/etherion_ai/graphql_schema/subscriptions.py`:

1. The resolver extracts the authenticated user from `request.state.auth_context` (populated by middleware for HTTP) or from `connection_params['headers']['Authorization']` (for WebSocket init).
2. It verifies that the user has access to the job (belongs to the same tenant, owns the job).
3. It calls `subscribe_to_job_status(job_id)`, which invokes the Redis client's subscription handler.

The resolver then creates an async generator that yields messages as they arrive on the Redis channel.

#### Step 4: Redis Pub/Sub Delivers Messages

As the worker executes the job, it calls `publish_execution_trace()` to emit detailed events:

```python
await publish_execution_trace(job_id, {
    "type": "SPECIALIST_REQUEST",
    "specialist_id": "research_specialist",
    "timestamp": datetime.utcnow().isoformat()
})
```

Redis fans this message to all active subscribers on the `job_trace_{job_id}` channel. The local subscription handler in the RedisClient picks up the message from the Pub/Sub stream and places it into an asyncio.Queue.

#### Step 5: Subscription Resolver Yields to Client

The resolver's async generator pulls from the queue and yields a `JobStatusUpdate` GraphQL object:

```python
async for evt in subscribe_to_execution_trace(job_id):
    yield JobStatusUpdate(
        job_id=evt.get("job_id"),
        status=evt.get("type"),
        message=evt.get("step_description"),
        timestamp=evt.get("timestamp"),
        additional_data=evt
    )
```

Strawberry serializes this to JSON and sends it over the WebSocket as a GraphQL subscription message.

#### Step 6: Frontend Client Updates UI

Apollo Client receives the subscription message on the WebSocket and dispatches it to the query cache. Any component listening to the `subscribeToJobStatus` or `subscribeToExecutionTrace` subscription receives the update. The job store (or React component) updates the UI:

- Progress bar advances to 45%.
- Execution timeline gains a new "SPECIALIST_REQUEST" node.
- Toast notifies the user: "Research specialist is analyzing results."

#### Step 7: Job Completes

The worker publishes a final status update:

```python
await publish_job_status(job_id, {
    "status": "COMPLETED",
    "message": "Job finished successfully",
    "progress_percentage": 100
})
```

The resolver receives this, yields the update, and upon seeing a terminal status, breaks out of the async loop and closes the subscription on the resolver side. The frontend receives the "COMPLETED" status, stops listening for updates, and shows the final results.

### 8.4 WebSocket Authentication: Different from HTTP

HTTP requests travel through the middleware stack, where headers are parsed and an `auth_context` is attached to `request.state`. However, WebSocket connections do not traverse the same middleware path in Strawberry's architecture.

#### How WebSocket Auth Works

1. **Connection Init Phase:**
   - The frontend (Apollo client) opens a WebSocket and immediately sends a `connection_init` message per the `graphql-ws` protocol.
   - The `connection_init` includes connection parameters, typically: `{ "headers": { "Authorization": "Bearer <JWT_TOKEN>" } }`.

2. **Resolver-Level Auth Check:**
   - Strawberry's `GraphQLRouter` receives the `connection_init` and passes `connection_params` to the context getter.
   - Inside the subscription resolver, the code checks two sources:
     - `request.state.auth_context` (if middleware already populated it for this connection).
     - `connection_params['headers']['Authorization']` (fallback, explicit for WebSockets).
   - The resolver calls `resolve_current_user_from_headers()` to decode the JWT and extract the user and tenant.

3. **Tenant Context Preservation:**
   - Once authenticated, the resolver **must** explicitly set the RLS tenant context for the database session.
   - In `subscribeToExecutionTrace`, this is done via:
     ```python
     set_session_tenant_context(session, current_user.tenant_id)
     ```
   - This ensures that any RLS-scoped database query executed within the subscription context returns only rows belonging to that tenant.

4. **Graceful Auth Failure:**
   - If the token is invalid or missing, the resolver yields a single `JobStatusUpdate` with `status="ERROR"` and `error_message="Authentication required"`, then terminates the stream.
   - No partial data is leaked.

#### Key Difference from HTTP

- **HTTP middleware** runs automatically for every request before the handler, attaching auth to the request state.
- **WebSocket subscriptions** bypass this pipeline; auth must be extracted inside the resolver and applied explicitly.
- This design ensures that each resolver can validate the requesting user independently, without relying on a shared middleware layer that may not apply to all connection types.

### 8.5 Redis as a Caching Layer Beyond Pub/Sub

While Redis's primary role in this system is Pub/Sub, it also serves as an application cache for performance and cost reduction:

#### Caching Helpers

The `src/core/redis.py` module provides namespaced get/set helpers:

```python
async def set(self, key: str, value: Any, expire: Optional[int] = None) -> bool:
    """Set a key-value pair with optional TTL."""
    await client.set(key, json_serialize(value), ex=expire)

async def get(self, key: str, default: Any = None) -> Any:
    """Retrieve a cached value."""
    return json_parse(await client.get(key, default=default))
```

#### Cached Object Types

The platform caches several long-lived or expensive-to-recompute objects:

1. **User Observation Context** (`context:{user_id}`):
   - The aggregated context (recent conversations, settings, preferences) built at session start.
   - TTL: 1 hour.
   - Avoids re-aggregating on every request.

2. **Tool Registry Snapshots** (`tools:registry:{version}`):
   - The list of deployed tools, their schemas, and availability status.
   - TTL: 30 minutes.
   - Reduces DB queries during tool validation and LLM prompt injection.

3. **Tenant Configuration** (`config:{tenant_id}`):
   - Tenant-level settings (model choice, KB search settings, cost limits).
   - TTL: 15 minutes.
   - Allows fast lookups without consulting the database on every request.

4. **Execution Cost Accumulation** (`cost:{job_id}:{tenant_id}`):
   - Running total of tokens and cost for an ongoing job.
   - TTL: 24 hours (survives job completion for archival).
   - Updated in real time as specialists and tools run.

#### Cache Invalidation

When a cached object is updated in the database, the cache is invalidated by deleting the key:

```python
await redis.delete(f"tools:registry:{version}")
```

If a bulk invalidation is needed (e.g., during evaluation reset), the `delete_by_pattern()` method is used:

```python
await redis.delete_by_pattern("teams:tenant:42:*")
```

#### Test-Friendly Mirroring

For testing and local development, the `RedisClient` maintains an in-memory mirror store (`_mirror_store`):

- Every `set()` operation also updates the mirror.
- Tests can introspect this mirror to assert cache behavior without needing a real Redis server.
- The mirror is not used for Pub/Sub (which only works via real Redis or a test double).

---

### 8.6 Resilience and Guarantees

#### Message Ordering

Events published to the same `job_id` channel are delivered in order because Redis Pub/Sub is a single-queue, FIFO stream. If event A is published before event B to `job_trace_{job_id}`, subscribers will receive them in that order.

#### At-Most-Once Delivery

Redis Pub/Sub does not persist messages. If a subscriber is not connected when a message is published, the message is lost. This is acceptable for UI updates because:
- The UI can tolerate missing intermediate events.
- Job status is queryable via the standard GraphQL query `query { job(id: "abc") { status } }` if a subscription update is missed.
- The execution trace is also archived in the database for later inspection.

#### Subscriber Limits

To prevent resource exhaustion from malicious or misbehaving clients, subscription resolvers enforce per-IP limits:

- Connection limit: max 10 concurrent WebSocket subscriptions per IP.
- Message rate limit: max 60 messages per minute per subscription stream per IP.

When a limit is exceeded, the resolver silently drops the excess message (for rate limiting) or rejects the connection (for connection limits).

4. **Update the frontend subscription document** (e.g., `frontend/lib/graphql-operations.ts`):

5. **Test end-to-end:**
   - Publish a test event from a worker or admin endpoint.
   - Verify the frontend receives it within 1 second.
   - Check that auth validation rejects cross-tenant access.

### 8.9 Common Pitfalls and Debugging

#### WebSocket Connections Closed Immediately

**Symptom:** "WebSocket connection failed" or "subscription failed" messages in the browser console.

**Causes:**
- Missing or invalid JWT in `connection_init` headers.
- Subscription resolver auth check failed; resolver emitted an ERROR payload and closed.
- Network firewall or reverse proxy blocking WebSocket upgrade headers.

**Debug steps:**
1. Check browser DevTools > Network > WS tab; look for the `/graphql` WebSocket.
2. Verify the `Authorization` header in the `connection_init` message (should be a valid JWT).
3. Check API logs for resolver-level auth errors.
4. Ensure `NEXT_PUBLIC_GRAPHQL_WS_ENDPOINT` is correctly configured and reachable.

#### Messages Not Arriving in Real Time

**Symptom:** Subscription connected but no events received; refresh or manual query shows the job is complete.

**Causes:**
- Events were published before the subscription was connected (Pub/Sub is fire-and-forget).
- Redis connection issue; `publish()` or `subscribe()` calls failing silently.
- Rate limiting triggered; excess messages dropped.

**Debug steps:**
1. Check Redis connectivity: `redis-cli ping` or via the API health check.
2. Log the `publish_*()` calls in the worker to confirm events are being sent.
3. Check resolver logs for rate limit messages (`"Too many messages..."`).
4. Query the job directly to see its current state (a subscription update may have been missed, but the state is still queryable).

#### High Redis Memory Usage

**Symptom:** Redis memory growing unbounded; eviction or OOM.

**Causes:**
- Cache keys accumulating without TTL (likely a bug in `set()` calls).
- Pub/Sub channel queues growing if subscribers are slow or disconnected abruptly.

**Debug steps:**
1. Use `MEMORY DOCTOR` and `KEYS *` commands to inspect memory distribution.
2. Ensure all cache `set()` calls include an `expire` parameter.
3. Monitor active subscription count; close idle WebSocket connections on the frontend.

## 9. Infrastructure Services and the etherion CLI

Etherion is a self-hosted platform that requires four backing services to function: PostgreSQL with pgvector, Redis, MinIO, and optionally HashiCorp Vault. Each service plays a distinct role in the platform's architecture, and the `etherion` CLI provides unified commands to initialize, manage, and operate these services alongside the platform itself.

### 9.1 PostgreSQL with pgvector

PostgreSQL serves as Etherion's primary data store, responsible for persisting all structured data including tenants, users, projects, conversations, job records, agent configurations, and execution logs. Additionally, PostgreSQL stores vector embeddings for the Knowledge Base using the `pgvector` extension, eliminating the need for a separate vector database and simplifying the operational footprint.

**Key Configuration:**

The PostgreSQL connection is specified via two environment variables that must point to the same database:
- `DATABASE_URL` for synchronous ORM operations (format: `postgresql+psycopg2://user:pass@host:5432/dbname`)
- `ETHERION_DATABASE_URL` for asynchronous operations (format: `postgresql+asyncpg://user:pass@host:5432/dbname`)

Both variables are required because Etherion uses both synchronous (for CLI operations and initial request handling) and asynchronous (for async workers and concurrent I/O) database access patterns. The actual database connection pooling and query execution differs between these drivers, but they connect to the same logical database.

**Tenant Isolation via Row Level Security:**

Row Level Security (RLS) is enforced at the database level through PostgreSQL policies on every table. This means the database itself—not just the application code—prevents a tenant from querying another tenant's data. When a query executes, PostgreSQL checks the current session's `app.current_tenant_id` setting and only returns rows matching that tenant's ID. This defense-in-depth approach ensures that even if an application bug allows an unauthorized query, the database rejects it.

**Schema Management:**

Alembic handles all database migrations. When you run `etherion migrate`, it executes `alembic upgrade head` to apply any pending schema changes. Migration files live in `etherion/_data/alembic/versions/` and are version-controlled, so schema changes are reproducible and reviewable before deployment.

### 9.2 Redis

Redis functions as Etherion's job broker, result backend, cache layer, and real-time messaging system. It runs on the default port 6379 and expects no authentication by default (though this can be changed in production deployments).

**Celery Job Processing:**

Redis stores the Celery job queue via `CELERY_BROKER_URL` and the job result backend via `CELERY_RESULT_BACKEND`. Jobs submitted by the API (such as document uploads to the Knowledge Base, scheduled agent runs, or long-running analyses) are serialized and pushed onto a Redis list. Celery workers pop from this queue, execute the job, and store results back in Redis for the API to retrieve asynchronously.

**Pub/Sub for Real-Time Updates:**

Etherion uses Redis Pub/Sub to broadcast job status updates (created, running, completed, failed) to connected WebSocket clients. When a job's state changes, a worker publishes a message to a Redis channel, and the API relays this to any client subscribed to that job's events. This can use the same Redis instance as the Celery broker or a separate logical database on the same server.

**General Caching:**

The cache layer stores tenant configurations, user observation metadata, and the tool registry. This reduces repeated database queries and improves response latency for frequently accessed data. Cache TTLs are configurable per data type.

### 9.3 MinIO (Object Storage)

MinIO is an S3-compatible object storage service that stores uploaded files, Knowledge Base source documents, repository assets, and AI-generated artifacts (reports, code snippets, etc.). Because MinIO speaks the S3 API, the same client code works seamlessly with Amazon S3 if you later decide to use AWS instead of self-hosting MinIO.

**Access Configuration:**

MinIO is configured via three environment variables:
- `MINIO_ENDPOINT` - the network address of the MinIO server (e.g., `minio:9000` or `192.168.1.50:9000`)
- `MINIO_ACCESS_KEY` - the access key ID (analogous to AWS Access Key ID)
- `MINIO_SECRET_KEY` - the secret key (analogous to AWS Secret Access Key)

**File Organization:**

Files are organized under tenant-specific prefixes to maintain isolation and simplify cleanup during tenant deletion:
- Knowledge Base source documents: `tenants/{tenant_id}/kb/{file_name}`
- Repository assets: `tenants/{tenant_id}/assets/{asset_id}`
- Generated artifacts: `tenants/{tenant_id}/artifacts/{artifact_id}`

**Integration Points:**

The `RepositoryService` handles uploads and downloads for repositories, while the `ContentRepositoryService` manages Knowledge Base documents. Both services use the MinIO client under the hood, abstracting away the connection details so business logic remains storage-agnostic.

### 9.4 Vault (Optional)

HashiCorp Vault provides centralized, encrypted storage for sensitive per-tenant credentials: OAuth tokens, API keys for third-party integrations, database credentials for data connectors, and other secrets. It is optional and is only used if `SECRETS_BACKEND=vault` is set in the environment.

**Alternative: Environment Variables:**

By default, `SECRETS_BACKEND=env`, which means secrets are read directly from environment variables. This is simpler for development but less secure than Vault for production, since environment variables are more difficult to rotate and audit. Choose `SECRETS_BACKEND=env` for test/staging environments and `SECRETS_BACKEND=vault` for production deployments handling sensitive customer data.

**Vault Path Convention:**

When a secret is requested, Vault queries the path `secret/{tenant_id}/{key_name}`. For example, a GitHub OAuth token for tenant `acme-corp` would be stored at `secret/acme-corp/github_oauth_token`. The `TenantSecretsManager` class handles all Vault interactions, providing a unified interface regardless of which backend is active.

### 9.5 The etherion CLI

The `etherion` CLI is the primary interface for initializing, bootstrapping, managing, and operating the platform. All commands are idempotent where possible and provide clear error messages if prerequisites are missing.

#### 9.5.1 Initialization: `etherion init`

**Usage:** `etherion init`

**What it does:** Sets up the project directory with all necessary configuration templates and database migration files.

**Specifically:**
- Copies `.env.example` to `.env` (or warns if `.env` already exists)
- Copies `docker-compose.services.yml` into the current directory
- Copies `alembic.ini` and `alembic/env.py` for database migration management

**When to use:** Run this exactly once when first deploying Etherion on a new machine or in a new environment. After `etherion init`, you edit `.env` to set credentials, database connections, API keys, etc.

**Prerequisites:** None. This is the first command you run.

#### 9.5.2 Bootstrapping Services: `etherion bootstrap`

**Usage:** `etherion bootstrap --mode docker` or `etherion bootstrap --mode native`

**What it does:** Starts the four backing services (PostgreSQL, Redis, MinIO, and optionally Vault) and verifies they are healthy.

**Mode: `--mode docker`**

Uses Docker Compose to start services in containers. It runs `docker compose up -d` with the `docker-compose.services.yml` file created by `etherion init`, waits for all containers to report healthy, and exits. This mode is recommended for development and is the default if no mode is specified.

**Mode: `--mode native`**

Skips Docker entirely and assumes the services are already running and managed externally (e.g., via systemd units, Kubernetes, or manual processes). The command polls each service's health endpoint to confirm connectivity. Use this mode when you have PostgreSQL, Redis, and MinIO installed directly on the host or coordinated by an orchestration layer.

**Polling and Health Checks:**

`etherion bootstrap` polls each service until it responds to a health check:
- PostgreSQL: attempts a connection and a simple `SELECT 1` query
- Redis: sends a PING command
- MinIO: requests the service health endpoint
- Vault: checks the health endpoint (if enabled)

If any service fails to become healthy within the timeout (typically 2-5 minutes), the bootstrap command exits with an error and a clear message indicating which service is not ready.

**When to use:** Run this after `etherion init` and before `etherion migrate`. It should also be run whenever you restart services or recover from a service failure.

#### 9.5.3 Database Migrations: `etherion migrate`

**Usage:** `etherion migrate`

**What it does:** Applies all pending database schema changes by running `alembic upgrade head`.

**Prerequisites:**
- `.env` must be populated with a valid `DATABASE_URL`
- The PostgreSQL database must be running and accessible
- The user/role specified in `DATABASE_URL` must have schema creation permissions

**Behavior:** Alembic compares the schema version recorded in the database (stored in the `alembic_version` table) against all migration files in `etherion/_data/alembic/versions/`. It applies each migration in sequence, updating the version table as it goes. If a migration fails, Alembic rolls back and reports the error; the database is left in a consistent state.

**When to use:** Run `etherion migrate` once after bootstrapping services, and again whenever you pull code changes that include new migrations (alembic files).

**Idempotency:** Running `etherion migrate` multiple times is safe; if the schema is already up to date, it will report "Current revision is already head" and exit cleanly.

#### 9.5.4 API Server: `etherion serve`

**Usage:** `etherion serve`

**What it does:** Starts the Etherion FastAPI application server, listening on a configurable port (default 8000).

**Prerequisites:**
- All four backing services must be running and healthy (confirmed via `etherion bootstrap`)
- The database must be migrated (confirmed via `etherion migrate`)
- `.env` must include `DATABASE_URL`, `REDIS_URL`, `JWT_SECRET_KEY`, and `SECRET_KEY`
- `LLM_PROVIDER` and the corresponding API key (e.g., `ANTHROPIC_API_KEY`, `GEMINI_API_KEY`, or `OPENROUTER_API_KEY`) must be set

**What it starts:**
- A Uvicorn ASGI server hosting the FastAPI application
- WebSocket endpoints for real-time job status updates and agent conversation streaming
- REST endpoints for all platform operations (tenants, users, projects, conversations, jobs, etc.)
- GraphQL endpoint (if `GRAPHQL_ENABLED=true`)

**When to use:** Run `etherion serve` in production and development. For production, run it under a process supervisor (systemd, supervisor, etc.) to auto-restart on crash.

**Configuration Defaults:**
- Port: 8000 (override with `ETHERION_PORT`)
- Workers: 1 (override with `ETHERION_WORKERS`)
- Host: 0.0.0.0 (override with `ETHERION_HOST`)

#### 9.5.5 Background Workers: `etherion worker start`

**Usage:** `etherion worker start`

**What it does:** Starts a Celery worker process that consumes jobs from Redis and executes them.

**Prerequisites:**
- Redis must be running (specified by `CELERY_BROKER_URL` and `CELERY_RESULT_BACKEND` in `.env`)
- PostgreSQL must be running and migrated
- `.env` must include `DATABASE_URL`

**What it processes:**
- Document uploads to the Knowledge Base (embedding, indexing, storage)
- Scheduled agent runs
- Long-running analyses and report generation
- Any other asynchronous job submitted by the API

**When to use:** Run one or more worker processes in production. A single worker handles jobs sequentially; for higher throughput, start multiple workers (typically one per CPU core). Each worker maintains its own database connection and Redis subscription, so they scale horizontally.

**Scaling:** To run 4 workers, start 4 separate processes (or use a supervisor like systemd or Docker to manage them):
```
etherion worker start &
etherion worker start &
etherion worker start &
etherion worker start &
```

#### 9.5.6 Tenant Management: `etherion create-tenant`

**Usage:** `etherion create-tenant --name "Acme Corp" --email "admin@acme.com" --password "SecurePassword123"`

**What it does:** Directly inserts a new Tenant record and an admin User into the database, bypassing the registration flow. Useful for bootstrapping the first tenant or providing offline tenant creation for special cases.

**Parameters:**
- `--name` - the display name of the tenant (e.g., "Acme Corp")
- `--email` - email address of the admin user
- `--password` - hashed and stored securely (using bcrypt)

**Output:** Prints the created tenant ID and user ID for reference.

**When to use:** During initial setup to create the first admin account, or when provisioning tenants programmatically without using the web UI.

#### 9.5.7 Health and Diagnostics: `etherion status`

**Usage:** `etherion status`

**What it does:** Checks connectivity to all configured services and displays a health summary table.

**Output Example:**
```
Service         Status      Version    Details
─────────────────────────────────────────────
PostgreSQL      UP          14.5       1 connection pool ready
Redis           UP          7.0        6379 responding
MinIO           UP          2024.01    etherion bucket exists
Vault           DOWN        N/A        Not configured or unreachable
FastAPI         DOWN        N/A        Not running on localhost:8000
Celery          UP          5.3        1 worker online
```

**When to use:** Run `etherion status` to verify the platform is healthy before debugging issues or after a system restart. It's also useful in monitoring scripts to confirm all services are ready.

#### 9.5.8 Platform Information: `etherion where`

**Usage:** `etherion where`

**What it does:** Shows a comprehensive summary of all Etherion-related files, processes, and services on the machine.

**Includes:**
- Location of `.env` and other config files
- Paths to database files (if PostgreSQL is running locally)
- Docker container names and IDs (if using Docker mode)
- Running Etherion processes (API server, workers)
- Active database connections and their owners
- Redis connection details
- MinIO bucket names and storage usage

**When to use:** Useful for troubleshooting, especially in complex deployments where multiple Etherion instances might be running, or when you need to locate log files or state directories.

#### 9.5.9 Version Information: `etherion version`

**Usage:** `etherion version`

**What it does:** Displays the installed version of the etherion package and verifies that all core components are importable and functional.

**When to use:** Quick check that the package is properly installed and not corrupted.

### 9.6 Environment Configuration (.env)

All Etherion configuration is managed through a single `.env` file in the project directory. This file is parsed by the CLI and API server at startup. Below is a structured reference of all configuration variables, grouped by function.

#### 9.6.1 Required Database Configuration

These variables are required for the API server and any database operation:

| Variable | Example | Purpose |
|----------|---------|---------|

#### 9.6.2 Required Redis Configuration

| `CELERY_RESULT_BACKEND` | `redis://localhost:6379/2` | (Optional) Separate Redis database for job results; defaults to `REDIS_URL` |

You can use the same Redis database for all three purposes or split them across different databases (0, 1, 2, etc.) or even different Redis instances for isolation and performance.

#### 9.6.3 Required MinIO Configuration

| `MINIO_SECURE` | `false` | Use TLS (true) or plain HTTP (false); default is false for local deployments |

For production, set `MINIO_SECURE=true` and use valid TLS certificates.

#### 9.6.4 Required Security Configuration

| Variable | Example | Purpose |
|----------|---------|---------|
| `SECRET_KEY` | (any 32+ character random string) | Used for secure session tokens and CSRF protection |
| `JWT_SECRET_KEY` | (any 32+ character random string) | Used for JWT token signing and verification |

Generate these with:

| `ANTHROPIC_API_KEY` | (your API key) | Required if `LLM_PROVIDER=anthropic` |
| `GEMINI_API_KEY` | (your API key) | Required if `LLM_PROVIDER=gemini` |

Only set the API key corresponding to your chosen provider.

| `VAULT_TOKEN` | (your token) | Vault authentication token; required if `SECRETS_BACKEND=vault` |

#### 9.6.7 Optional Knowledge Base Configuration

| Variable | Example | Purpose |

| `KB_EMBEDDING_DIM` | `1536` | Dimensionality of embeddings; typically 1536 for OpenAI models, 768 for smaller models |
| `KB_EMBEDDING_MODEL` | `text-embedding-3-small` | Model to use for generating embeddings (must match your LLM provider's capabilities) |

| `ETHERION_WORKERS` | `4` | Number of Uvicorn worker processes; default is 1 |
| `ETHERION_RELOAD` | `false` | Auto-reload on code changes (development only); default is false |

| Variable | Example | Purpose |
|----------|---------|---------|
| `LOG_LEVEL` | `INFO` | Minimum log level: DEBUG, INFO, WARNING, ERROR, CRITICAL; default is INFO |
| `LOG_FORMAT` | `json` or `text` | Log output format; default is text |

#### 9.6.10 Optional Deployment Configuration

| Variable | Example | Purpose |
|----------|---------|---------|
| `ENVIRONMENT` | `development` or `production` | Environment name; affects debugging and error reporting |
| `ALLOWED_ORIGINS` | `http://localhost:3000,https://app.example.com` | CORS allowed origins; comma-separated list |

**Production Best Practices:**

1. **Secrets:** Store `SECRET_KEY`, `JWT_SECRET_KEY`, and API keys in Vault or a secrets management system, not in `.env`. If you must use `.env`, ensure the file has restricted permissions (`chmod 600 .env`) and is not committed to version control.

2. **Database:** Use a dedicated PostgreSQL instance (not the Docker container) for production. Configure backups and replication.

3. **Redis:** Use a managed Redis service or a high-availability Redis cluster with persistence enabled.

4. **MinIO:** Use a distributed MinIO cluster with redundancy and regular backups.

5. **Logging:** Set `LOG_FORMAT=json` for structured logging, which integrates well with log aggregation services.

6. **Monitoring:** Periodically run `etherion status` in a cron job or monitoring service to confirm all backing services remain healthy.

## 10. The Etherion Terminal UI

### 10.1 What the Terminal UI Is

The Etherion Terminal UI (`etherion-tui`) is a full-featured, cross-platform terminal user interface built with Go using the Bubble Tea framework from charmbracelet. It provides a rich, interactive way to manage Etherion deployments, create and run jobs, monitor executions, and configure integrations—all from the command line.

Unlike standalone CLI commands, the TUI offers a persistent, navigable interface with eight purpose-built tabs that work together as a cohesive system. The TUI connects to the same HTTP and WebSocket APIs as the web browser interface, meaning it provides feature parity with the web application while delivering a more direct, keyboard-driven workflow suitable for operators, developers, and AI engineers working in terminal-based environments.

### 10.2 Distribution and Installation

**Installation**: The TUI is distributed as a Python package (`pip install etherion-tui`) that bundles pre-compiled Go binaries for all major platforms. This single `pip` command handles platform detection and binary setup, making it simple to install alongside the main `etherion` CLI.

**Supported Platforms**:
- Linux x86_64
- Linux ARM64
- Windows x86_64
- macOS Intel (x86_64)
- macOS Apple Silicon (ARM64)

**How It Works**: When a user runs `etherion-tui`, the Python entrypoint (`tui-pkg/etherion_tui/__init__.py:main()`) determines the platform and architecture using `platform.system()` and `platform.machine()`, selects the appropriate pre-compiled binary from the package's `bin/` directory, sets the executable bit on Unix systems, and then uses `os.execv()` to replace the Python process with the Go binary. On Windows, it uses `subprocess.call()` instead. This approach ensures that end-users never interact with Python—they launch a native, statically-linked binary.

### 10.3 Core Architecture and Navigation

The TUI is a single-window terminal application with eight tabs, each managing a specific domain of functionality. The tabs are presented as a horizontal menu at the top, and pressing keys 1–8 jumps directly to any tab. Tab/Shift+Tab cycles through them sequentially. The active tab's content fills the rest of the screen.

**Top-Level Model** (`tui/internal/ui/model.go`):
- `RootModel` is the main Bubble Tea model. It owns the current active tab index, terminal dimensions, configuration (loaded from `~/.config/etherion/tui.json`), an API client (if logged in), and a sub-model for each of the eight tabs.
- On startup, if the config contains a stored JWT access token, the TUI initializes with the logged-in user and jumps to the Dashboard tab. Otherwise, it starts on the Connect tab.
- The model runs a background health-check every 3 seconds for the Dashboard tab.

**Global Key Bindings**:
- `Ctrl+C`: quit immediately from anywhere
- `1–8`: jump to the corresponding tab
- `Tab` / `Shift+Tab`: cycle through tabs
- `Q`: quit (disabled on Connect and Setup tabs to prevent accidental exit during forms)

### 10.4 The Eight Tabs

#### **Tab 1: Connect**

**Purpose**: Log in to an existing Etherion deployment or register the first admin account.

**Form Fields**:
- **Server URL** (default: `http://127.0.0.1:8080`): The HTTP base URL of the Etherion API.
- **Email**: User email address.
- **Password**: User password (displayed as masked input).

**Modes**:
- **Login mode** (default): Send credentials to the API's `/graphql` login mutation. On success, the API returns a JWT access token, which the TUI stores in `~/.config/etherion/tui.json` and switches to the Dashboard tab.
- **Register mode** (press `Ctrl+R`): Create the first admin account for a fresh deployment. In this mode, pressing Enter executes `etherion create-tenant` locally (without contacting the server) to provision a new tenant and admin user, then saves the auth credentials and server URL to config, and switches to the Dashboard.

**Use Cases**:
- First-time setup: press `Ctrl+R` to create an admin account on a freshly bootstrapped system.
- Multi-user access: use Login mode to connect existing users to the deployment.
- Server switching: change the Server URL and re-login to connect to a different Etherion instance.

#### **Tab 2: Setup**

**Purpose**: Onboarding wizard for new deployments. Guides operators through initializing Etherion services, running migrations, creating a tenant, and launching the platform.

**Flow Overview**:
1. **Mode Selection** (`sStepIdle`): Choose between Docker (uses `docker compose up -d` to start services) or Bare Metal (services are expected to be running externally via systemd or similar).
2. **Binary Verification** (`sStepVerify`): The wizard calls `config.EtherionBin()` to locate the `etherion` CLI binary. This uses a five-step resolution order (see §10.6 below).
3. **Environment Check** (`sStepCheckEnv`): Verifies that a `.env` file exists in the current directory or the `$ETHERION_HOME` directory.
4. **Bootstrap** (`sStepBootstrap`, Docker mode only): Runs `etherion bootstrap` to start Docker Compose services (PostgreSQL, Redis, MinIO, API, Worker).
5. **Migrations** (`sStepMigrate`): Runs `etherion migrate` to apply database schema migrations.
6. **Tenant Creation** (`sStepTenantForm` → `sStepTenant`): A two-field form (email and password) collects admin credentials. On submit, the wizard runs `etherion tenant create EMAIL PASSWORD` locally.
7. **OAuth Setup** (`sStepOAuth`): Manual step—user is instructed to connect OAuth providers via the OAuth tab (no automated setup in the wizard).
8. **Server Launch** (`sStepServe`): Runs `etherion serve` as a detached background process.
9. **Done** (`sStepDone`): Wizard completes; user is advised to check the Logs tab for any errors.

**Live Output**: Each step displays real-time stdout/stderr from the executed command, allowing the user to see progress and diagnose issues immediately.

**Retry Logic**: If a step fails (non-zero exit code), the wizard marks `lastFailed = true` and waits for the user to press Enter to retry that step.

**Use Cases**:
- Deploying Etherion in a Docker-local environment (development machine, CI/CD runner).
- Setting up a bare-metal deployment where services are managed via systemd, Kubernetes, or other orchestrators.
- Troubleshooting: the live output from each step helps diagnose configuration or dependency issues.

#### **Tab 3: Chat**

**Purpose**: Conversation interface for running AI jobs against projects.

**Interaction Model**:
- Select a project from a dropdown.
- Select or create a conversation thread.
- Type a goal (the user's request to the AI).
- Press Enter to start a job.
- Watch the live execution trace as agents work (via WebSocket subscription to `subscribeToExecutionTrace`).

**Display**:
- Conversation thread history above.
- Input field at the bottom.
- Execution trace streamed in the center as the job runs.

**Use Cases**:
- Running AI research, analysis, or coding tasks.
- Debugging agent behavior by observing the trace in real time.
- Rapidly iterating on prompts and agent teams without leaving the terminal.

#### **Tab 4: Agents**

**Purpose**: Browse, create, and manage Custom Agent Definitions and Agent Teams.

**Capabilities**:
- List all Agent Definitions and Teams for the current tenant.
- Create new Custom Agent Definitions by specifying name, instructions, and tools.
- Create new Agent Teams by composing multiple agents with roles and capabilities.
- View and edit existing definitions and teams.
- Mark teams as system-owned (read-only) or user-created (editable).

**Data Model**: Definitions and teams are persisted to the database and loaded via GraphQL queries. The TUI's Agents tab provides a guided UI for CRUD operations rather than requiring direct API calls.

**Use Cases**:
- Experimenting with different agent configurations.
- Curating reusable agent templates for a tenant.
- Managing role-based agent teams for specific workflows.

#### **Tab 5: Monitor**

**Purpose**: Job history and execution summary table.

**Display**:
- List of past jobs, sorted by creation date (most recent first).
- Columns: job ID, status (pending, running, succeeded, failed), start/end time, duration, cost.
- Quick access to trace summaries for each job.

**Filtering and Search**: Navigate the table with arrow keys, press Enter to view details or fetch the full trace.

**Use Cases**:
- Checking job status and duration (useful for cost tracking and performance tuning).
- Reviewing past runs to find successful patterns or debug failures.
- Accessing archived traces without re-running jobs.

#### **Tab 6: OAuth**

**Purpose**: Manage OAuth integrations and access tokens for third-party services (Google Drive, GitHub, Slack, etc.).

**Display**:
- List of available OAuth providers (e.g., `google_drive`, `github`, `slack`).
- For each provider: current connection status (connected/disconnected), connected account email or username.
- Actions: connect (opens a browser with an OAuth auth code flow), disconnect (revokes the token), or refresh (renew expiring tokens).

**How It Works**:
- When the user selects "Connect" for a provider, the TUI launches the default browser with an auth code flow redirect URI (`http://localhost:port/oauth/callback` or similar).
- The TUI listens on a local port, captures the authorization code, exchanges it for an access token via the Etherion API, and stores it securely in the platform's token vault.
- The TUI then displays the result and updates the connection status.

**Use Cases**:
- Setting up integrations for MCP tools (e.g., Google Workspace adapter needs Google OAuth).
- Managing token expiration and refresh without leaving the terminal.
- Viewing which external services a tenant has authorized.

#### **Tab 7: Logs**

**Purpose**: Real-time scrollable log viewer.

**Display**:
- Continuous stream of log lines from the API server or local CLI processes.
- Each line is prefixed with timestamp and log level (DEBUG, INFO, WARN, ERROR).
- Scroll up/down to view historical logs; logs are buffered in memory.

**Sources**:
- Logs from the running `etherion serve` process (when launched via the Setup tab).
- Logs streamed from the API server via a GraphQL subscription or server-sent events (SSE).

**Use Cases**:
- Debugging configuration or runtime issues.
- Monitoring background job execution.
- Auditing API activity (authentication, authorization, mutations).

#### **Tab 8: Dashboard**

**Purpose**: Health dashboard showing connectivity status for all platform services.

**Services Monitored**:
- **PostgreSQL**: TCP connectivity check to the database host and port.
- **Redis**: TCP connectivity check to the Redis host and port.
- **MinIO**: TCP connectivity check to the MinIO endpoint.
- **API Server**: HTTP GET to the API's `/health` endpoint.

**Display**:
- Service name, status icon (green checkmark for OK, red X for failure), response time (in milliseconds).
- Last check timestamp at the bottom.
- Services refresh every 3 seconds (configurable).

**Environment Sourcing**: Health checks read service URLs from environment variables (`DATABASE_URL`, `REDIS_URL`, `MINIO_ENDPOINT`) or fall back to standard localhost defaults (e.g., `localhost:5432` for PostgreSQL).

**Use Cases**:
- Quick verification that all services are reachable (before running jobs or in Setup wizard).
- Diagnosing network or service-down issues.
- Confirming that a fresh deployment is fully operational.

### 10.5 Setup Wizard Deep Dive

The Setup tab (Tab 2) walks a new operator through a complete fresh Etherion deployment in a guided, step-by-step manner. This section details the wizard's execution flow and the commands it runs.

**Initial Mode Choice**:
- The wizard begins by asking the user to choose between Docker or Bare Metal.
- **Docker**: Uses `docker compose` to orchestrate PostgreSQL, Redis, MinIO, API, and Worker containers.
- **Bare Metal**: Assumes services are already running (e.g., via systemd) and skips the Docker bootstrap step.

**Step 1: Find Etherion Binary**

The wizard resolves the `etherion` CLI binary using the five-step resolution order (see §10.6):
1. Config file override (`etherion_bin` field in `~/.config/etherion/tui.json`).
2. Environment variable (`ETHERION_BIN`).
3. Standard PATH lookup (`which etherion` or `where etherion` on Windows).
4. Same directory as the TUI binary (common with pip installs).
5. `~/.local/bin/etherion` (common pip `--user` install location).

If the binary is not found, the wizard fails and advises the user to install `etherion` via `pip install etherion` or set the `ETHERION_BIN` environment variable.

**Step 2: Check Environment File**

The wizard looks for a `.env` file in the current working directory or `$ETHERION_HOME`. This file should contain:
- Database URL (PostgreSQL connection string)
- Redis URL
- MinIO endpoint and credentials
- API server configuration (port, JWT secret, etc.)
- Optional: GCP project IDs, service account paths, third-party API keys

If the file is missing, the wizard fails and instructs the user to create it. A sample `.env.example` is typically provided in the Etherion repository.

**Step 3: Bootstrap Services (Docker Only)**

For Docker mode, the wizard runs:
```bash
etherion bootstrap
```

This command reads the `.env` file and executes `docker compose up -d`, starting all required services. The wizard displays stdout/stderr in real time so the user can see each container starting.

For Bare Metal mode, this step is skipped entirely.

**Step 4: Run Migrations**

The wizard runs:
```bash
etherion migrate
```

This applies all pending database schema migrations using Alembic (or similar). The wizard waits for migrations to complete and shows any errors.

**Step 5: Create Admin Tenant**

A two-field form appears:
- Email: email address for the first admin user.
- Password: password (masked input).

The wizard runs:
```bash
etherion tenant create EMAIL PASSWORD
```

This command:
- Creates a new tenant in the database.
- Creates an admin user under that tenant.
- Seeds any default system agents or teams.

The wizard saves the email and server URL to `~/.config/etherion/tui.json`.

**Step 6: OAuth Setup (Manual)**

The wizard displays a message instructing the user to switch to the OAuth tab (Tab 6) and connect any required OAuth providers (Google, GitHub, Slack, etc.). This is not automated because OAuth setup often requires external configuration (client IDs, redirect URIs in the provider's console).

**Step 7: Launch API Server**

The wizard runs:
```bash
etherion serve
```

This starts the API server (FastAPI + GraphQL endpoint) and the Worker process (Celery or similar job runner). The wizard launches this as a detached background process (using Unix `nohup` or Windows `START /B` depending on platform), so it does not block the TUI.

**Step 8: Done**

The wizard completes with a success message. The user is advised to:
- Switch to the Dashboard tab to verify all services are healthy.
- Switch to the Chat tab to run a test job.
- Switch to the Logs tab if any errors appear during setup.

### 10.6 Binary Resolution and Platform Packaging

The TUI needs the `etherion` CLI binary to be available for the Setup wizard (Steps 1, 3–5 above) and for the Register mode in the Connect tab. The binary resolution strategy ensures that users can install both the `etherion` CLI and the TUI via `pip install` without manual configuration, while still allowing override paths via environment variables or config files.

**Resolution Order** (in `tui/internal/config/config.go::Config.EtherionBin()`):

1. **Config File Override**: Check for `etherion_bin` field in `~/.config/etherion/tui.json`. If present and the file exists, use it.

2. **Environment Variable**: Check `ETHERION_BIN`. If set and the path exists, use it.

3. **PATH Lookup**: Call `exec.LookPath("etherion")` (or `etherion.exe` on Windows). On Unix systems, this searches the directories in `$PATH`; on Windows, it appends `.exe` automatically.

4. **Same Directory as TUI Binary**: On Unix, get the current process's executable path via `os.Executable()`, then look for `etherion` in that directory. On Windows, look for `etherion.exe`. This handles the common case where `pip install etherion etherion-tui` places both binaries in the same bin directory (e.g., `~/.local/bin/` or `C:\Program Files\Python\Scripts\`).

5. **User Local Bin**: Check `~/.local/bin/etherion` (Unix) or `%LOCALAPPDATA%\Python\Scripts\etherion.exe` (Windows). This is the default location for pip `--user` installs.

If none of these paths yield a valid file, the resolution returns an empty string and a `false` flag, and the Setup wizard fails gracefully with an error message.

**Platform-Specific Handling**:
- On Unix (Linux, macOS), the TUI's Python entrypoint sets the executable bit on the binary before exec'ing it.
- On Windows, the TUI's Python entrypoint uses `subprocess.call()` instead of `os.execv()` because Windows does not have an exec-like syscall.

**Cross-Platform Binaries**:

The TUI is distributed with four pre-compiled Go binaries, one for each platform:
- `etherion-tui-linux-amd64`
- `etherion-tui-linux-arm64`
- `etherion-tui-darwin-amd64` (macOS Intel)
- `etherion-tui-darwin-arm64` (macOS Apple Silicon)
- `etherion-tui-windows-amd64.exe`
- `etherion-tui-windows-arm64.exe`

These binaries are built at release time using `go build` with GOOS and GOARCH flags, and are bundled into the Python package's `tui-pkg/etherion_tui/bin/` directory. The binary names encode the platform and architecture so that the Python entrypoint can select the correct one.

**Build and Release Process** (outlined in build scripts):

1. For each target platform (Linux x86_64, Linux ARM64, macOS Intel, macOS Apple Silicon, Windows x86_64, Windows ARM64):
   - Run `GOOS=<os> GOARCH=<arch> go build -o etherion-tui-<os>-<arch>[.exe] ./tui/main.go`
   - Strip debug symbols (optional, to reduce binary size).
   - Sign or notarize (for macOS, optional).

2. Place all binaries in `tui-pkg/etherion_tui/bin/`.

3. Update `tui-pkg/setup.py` or `pyproject.toml` to include the binaries in `package_data` or `[tool.poetry.exclude]`.

4. Run `pip install -e tui-pkg` (editable) or build a wheel (`pip wheel tui-pkg`).

5. Test the TUI on each platform to verify binary selection and execution.

## 11. Security

Etherion implements a layered security architecture where multiple independent security mechanisms work together to protect user data, enforce tenant isolation, and defend against attacks. This multi-layer approach follows defense-in-depth principles: if one layer is compromised or bypassed, additional layers remain effective.

### 11.1 Layered Security Model

The security stack operates as follows, with each layer protecting against distinct threat vectors:

1. **Rate Limiting** blocks brute-force and denial-of-service attacks before they reach application logic
2. **CSRF Protection** prevents state-changing operations initiated from cross-origin sites
3. **Authorization** ensures authenticated users have permission for their requested operations
4. **Row-Level Security (RLS)** enforces tenant isolation at the database layer, making data leakage impossible even if middleware is bypassed
5. **Input Validation and Injection Detection** prevent malformed or malicious inputs from reaching the AI engine
6. **Audit Logging** records security events for forensic analysis and compliance

Each layer is independent and fail-closed: requests are rejected unless they pass all checks. The database layer provides the hardest guarantee—even if an attacker bypasses middleware, PostgreSQL's row-level security policies prevent cross-tenant data access.

### 11.2 Rate Limiting: Threshold-Based Defense

**Purpose:** Prevent brute-force authentication attempts, API flooding, and denial-of-service attacks.

**Implementation:** All incoming requests are subject to per-IP rate limits tracked in Redis using a sliding window algorithm. Limits are differentiated by endpoint type:

- Authentication endpoints: 10 requests/minute, 100 requests/hour
- GraphQL mutations: 30 requests/minute, 500 requests/hour
- GraphQL queries: 100 requests/minute, 1000 requests/hour
- Default endpoints: 60 requests/minute, 1000 requests/hour

When a client exceeds the per-minute limit, the server responds with HTTP 429 (Too Many Requests) and includes a `Retry-After` header. The rate limit state is stored in Redis with automatic expiration, so limits reset cleanly without manual intervention.

**Attack prevented:** Credential stuffing (thousands of login attempts against a single account), distributed brute-force attacks, and resource exhaustion via API flooding.

### 11.3 CSRF Protection: Double-Submit Cookie Pattern

**Purpose:** Prevent cross-site request forgery, where malicious websites trick users into initiating state-changing operations on Etherion.

**Implementation:** Etherion uses the double-submit cookie pattern. A CSRF token is generated per user session and embedded in both an HTTP cookie and a request header. For state-changing operations (POST, PUT, DELETE), the middleware validates that both the cookie and header are present and match before allowing the request.

- Token generation uses cryptographic HMAC-SHA256 with server-side secret validation
- Tokens include user ID, session ID, and timestamp; tokens expire after 1 hour
- Bearer token authentication and requests from non-browser clients are exempt, as they cannot be forged by cross-site scripts
- Protected paths include all non-GET requests except `/auth/`, `/api/tui/auth/`, GraphQL (which has its own CSRF guard), and webhooks

**Attack prevented:** Malicious websites cannot trigger account deletion, configuration changes, or other mutations on behalf of logged-in users.

### 11.4 Security Headers: Browser-Level Protections

**Purpose:** Instruct browsers to enforce security policies and prevent injection attacks.

**Implementation:** All responses include comprehensive security headers:

- **Content-Security-Policy (CSP):** Restricts which scripts and resources the browser may load, preventing inline script injection and reducing XSS impact
- **X-Frame-Options: DENY:** Prevents clickjacking by disallowing the site from being embedded in iframes
- **X-Content-Type-Options: nosniff:** Prevents browsers from MIME-sniffing and executing unexpected content types
- **Strict-Transport-Security (HSTS):** In production, enforces HTTPS for all future connections, preventing protocol downgrade attacks
- **Referrer-Policy:** Controls what information is leaked when navigating away from the site
- **Permissions-Policy:** Disables access to powerful browser APIs (geolocation, camera, microphone) that are not needed by the web UI

These headers are set on all responses, including error responses, with production and development variants.

**Attack prevented:** Cross-site scripting (XSS), clickjacking, man-in-the-middle protocol downgrades, and abuse of browser APIs.

### 11.5 Authorization: Role-Based Access Control

**Purpose:** Ensure users can only perform operations they are authorized for.

**Implementation:** Each request carries an Authorization header with a Bearer token. The token is validated and decoded to extract the user ID and tenant ID. The authorization middleware then checks:

1. **Authentication:** Valid JWT token signed with the server's secret key
2. **User existence:** The user exists in the database
3. **Tenant existence:** The user's tenant exists and is active
4. **Permissions:** For sensitive operations like tenant creation or user management, an Admin role check is performed

The authorization context is populated before processing the request, allowing resolvers and middleware to access `user_id`, `tenant_id`, and `role` from the request state.

**Attack prevented:** Unauthorized operations (a regular user cannot delete tenants), token reuse after logout (tokens are short-lived), and privilege escalation.

### 11.6 Row-Level Security: Database-Enforced Tenant Isolation

**Purpose:** Provide an absolute guarantee that one tenant's data cannot leak to another tenant, even if application-level authorization is bypassed.

**Implementation:** PostgreSQL Row-Level Security (RLS) is the primary isolation mechanism. Every table with multi-tenant data includes a `tenant_id` column with an RLS policy. Before any operation, the middleware sets:

```sql
SET app.tenant_id = <user_tenant_id>
```

on the database session. RLS policies on each table then filter rows to only those where `tenant_id = current_setting('app.tenant_id')`. For example, a SELECT on the `Job` table:

```sql
WHERE tenant_id = current_setting('app.tenant_id')::INTEGER
```

If an application bug allows a query to omit the tenant_id filter, the database layer still enforces it. A query that retrieves all jobs—even with no WHERE clause—only returns rows for the current tenant. The RLS policy is transparent to the application but guaranteed by the database.

**Additional guarantee:** If a developer accidentally writes:
```python
session.query(Job).filter(Job.id == job_id).first()
```

without filtering by tenant, the RLS policy silently adds the tenant filter, returning None if the job belongs to a different tenant. This prevents accidental cross-tenant data leakage.

**Attack prevented:** Tenant isolation breach, cross-tenant data access, lateral movement between customer accounts. RLS remains effective even if a middleware component is disabled or an attacker gains database access directly.

### 11.7 AI-Specific Security: Prompt Injection and Tool Control

**Purpose:** Prevent attackers from hijacking the AI orchestrator and executing unauthorized operations.

**Implementation:** Two distinct threats are mitigated:

**Prompt Injection Detection:** User inputs are scanned for prompt injection attempts—instructions designed to override the LLM's original task. The detector uses a combination of:

- Heuristic pattern matching for common injection keywords ("ignore previous instructions", "forget", "disregard")
- Configurable risk scoring with thresholds for sanitization (score ≥ 40) or blocking (score ≥ 80)
- Optional AI-based refinement when scores are near thresholds, using a lightweight LLM classifier
- Allowlisting of trusted users to skip blocking (though events are still audited)

When an injection is detected at or above the sanitize threshold, dangerous phrases are neutralized (replaced with `[redacted]`). When an injection reaches the block threshold, the input is rejected outright.

**Tool Execution Control:** The orchestrator implements a fail-closed policy for tool execution:

- Every tool must be registered in the tool registry before it can be called
- Every tool must be explicitly pre-approved by the team before it can be invoked during task execution
- Tool invocations are validated against the approval list during execution
- Custom agents and agent teams declare their pre-approved tools up-front; any tool not in this list is blocked

Write operations (tools that modify state) require explicit user confirmation via a confirm-action gate before execution.

**Attack prevented:** Prompt injection to steal secrets or exfiltrate data, tool chaining to perform unintended operations, supply-chain attacks via malicious tool definitions.

### 11.8 Audit Logging: Security Event Transparency

**Purpose:** Provide a complete record of security-relevant events for forensics, compliance, and anomaly detection.

**Implementation:** All security events are logged to both file (with rotation) and Redis (for real-time monitoring). Events include:

- **Authentication events:** Login success, login failure (reason, IP address, user agent)
- **Authorization failures:** Access denied, permission checks, cross-tenant access attempts
- **Security violations:** CSRF attempts, XSS attempts, SQL injection attempts, prompt injection blocks
- **Data access:** Who accessed which data, when, and from which IP
- **Rate limit hits:** Threshold exceeded, client IP, endpoint, retry-after time
- **Orchestrator operations:** Tool execution, confirmation gates, validation results

Each event is a structured JSON record with:
- Unique event ID (SHA-256 hash of content for idempotency)
- Timestamp (UTC ISO format)
- Severity level (low, medium, high, critical)
- User ID and tenant ID (if applicable)
- IP address (hashed for privacy in some contexts)
- User agent
- Detailed context specific to the event type

Audit logs are retained for 90 days by default (configurable) and rotated when they exceed 100 MB. Real-time events are published to Redis channels (`audit:events`, `audit:critical`, `audit:tenant:<tenant_id>`), allowing monitoring systems to subscribe and alert on suspicious activity.

**Why it matters:** Audit logs enable security investigation, compliance audits (SOC 2, GDPR), anomaly detection (e.g., unusual access patterns), and insider threat monitoring. In self-hosted deployments, audit logs are the organization's primary evidence of what occurred and when.

### 11.9 Defense in Depth: Layering in Practice

An example request flow illustrates how layers work together:

1. **Request arrives** from client IP
2. **Rate limiting** checks if the IP has exceeded per-minute threshold; if so, reject with 429
3. **CSRF protection** validates the CSRF token in the header matches the session cookie; if not, reject with 403
4. **Authorization** decodes the JWT and checks the user has permission for the operation; if not, reject with 401/403
5. **Middleware sets tenant context** via `SET app.tenant_id = <user_tenant_id>`
6. **Application queries database** with a filter like `WHERE id = ?`
7. **Database RLS policy** transparently adds `AND tenant_id = <current tenant>` to the query
8. **Database returns only the user's tenant's data** even if the query had no explicit tenant filter
9. **For AI operations**, inputs are scanned for injection; tools are checked against approval list; operations are logged

If an attacker compromises a single layer, the remaining layers provide additional barriers. If a CSRF token is leaked, rate limiting and authorization still protect the account. If authorization middleware is disabled, RLS ensures data isolation. If the database is directly accessed, RLS prevents cross-tenant queries.

### 11.10 Self-Hosted Security Considerations

Because Etherion is self-hosted on Matchbox (NixOS via Ansible) with local PostgreSQL, MinIO, and Vault:

- **Network isolation:** The platform runs on internal infrastructure; external access is controlled by the organization
- **Vault integration:** When `SECRETS_BACKEND=vault`, OAuth tokens and API keys are encrypted in transit and at rest using Vault's transit engine
- **Audit logs:** Stored locally on the filesystem and Redis; the organization maintains these logs and can integrate with SIEM systems
- **Database backups:** RLS policies are part of the schema and are automatically included in backups; restoring from backup preserves tenant isolation

Organizations should:
- Regularly review audit logs for suspicious activity
- Back up the database and MinIO buckets to isolated storage
- Restrict database access to the application server and authorized administrators
- Enable PostgreSQL query logging for high-security environments
- Monitor Vault for unauthorized access attempts

### 11.11 Threat Model Summary

| Threat | Primary Defense | Secondary Defense |
|--------|-----------------|-------------------|
| Brute-force authentication | Rate limiting (10/min) | JWT expiration (24h) |
| CSRF attacks | Double-submit cookie + Origin validation | Same-site cookie |
| Cross-tenant data leak | RLS policies (database-enforced) | Authorization middleware |
| XSS injection | CSP headers, input sanitization | Audit logging for detection |
| Unauthorized API access | Authorization middleware | Bearer token validation |
| Prompt injection | Pattern detection + risk scoring | Tool approval gate |
| Unauthorized tool execution | Registry + pre-approval check | Confirm-action gate, audit log |
| Insider credential abuse | Audit logging, API key expiration | Vault encryption |

No single layer is sufficient to protect the platform. The combination of rate limiting, CSRF protection, authorization, RLS, input validation, and audit logging creates a resilient system where defenders have multiple opportunities to detect and block attacks, and multiple locations to detect and investigate security events.

## 12. Configuration, Secrets, and LLM Providers

### Overview

Etherion AI configuration follows a "fail-early" philosophy: all runtime settings live in a single `.env` file at the project root, loaded at startup via python-dotenv. The CLI validates required variables before starting each service, reporting exactly which variable is missing and how to fix it. This prevents silent failures and eliminates ambiguity about what a deployment lacks.

The platform supports pluggable LLM providers (Gemini, Anthropic, OpenRouter) via a provider abstraction layer, allowing you to swap providers by changing a single environment variable. Secrets are stored either in environment variables or in HashiCorp Vault, depending on your deployment model.

---

### The .env File and Loading Strategy

All runtime configuration lives in a `.env` file (or environment variables). The Etherion CLI provides the template:

```bash
cp etherion/_data/env/.env.example .env
```

Edit the resulting `.env` file with your values:

- Database connection strings (sync and async)
- Redis URLs (general, async, Celery broker/result backend)
- Object storage credentials (MinIO or S3)
- API host and port
- Authentication keys (JWT, CSRF)
- LLM provider credentials
- Secrets backend choice (env or Vault)
- OAuth client credentials

**Loading sequence:**

1. On startup, `etherion/cli/_env.py` calls `require_dotenv()` to load `.env` into memory
2. If `.env` is absent, the CLI prints a clear error with actionable guidance
3. The loaded environment is merged over `os.environ`, so exported variables override `.env` values
4. `python-dotenv` is applied via `load_dotenv(".env", override=False)` in service entry points

**Environment variable precedence (highest to lowest):**

1. Process environment (e.g., `export DATABASE_URL=...` before running)
2. `.env` file values
3. Hardcoded defaults (rare, only for optional settings)

---

### Required Configuration Groups

#### Database (ORM & Async Handlers)

```
DATABASE_URL=postgresql+psycopg2://user:pass@localhost:5432/etherion
ETHERION_DATABASE_URL=postgresql+psycopg2://user:pass@localhost:5432/etherion
```

**What they control:**

- `DATABASE_URL`: Synchronous connection used by SQLAlchemy ORM for migrations and schema operations
- `ETHERION_DATABASE_URL`: Can be async (psycopg2/psycopg[asyncio] prefix) for FastAPI request handlers; in single-process dev, both can be identical

**If missing:**

Commands `migrate`, `serve`, and `worker start` fail immediately with:

```
✗ Missing required environment variable(s):
  • DATABASE_URL
```

The application does not attempt to proceed; this is by design.

#### Redis (Caching, Pub/Sub, Task Queue)

```
REDIS_URL=redis://localhost:6379/0
ETHERION_REDIS_URL=redis://localhost:6379/0
CELERY_BROKER_URL=redis://localhost:6379/0
CELERY_RESULT_BACKEND=redis://localhost:6379/1
```

**What they control:**

- `REDIS_URL`: General-purpose cache and pub/sub (e.g., session data, rate limits)
- `ETHERION_REDIS_URL`: Async Redis client for real-time features (WebSocket subscriptions, concurrent requests)
- `CELERY_BROKER_URL`: Task queue broker (which Redis DB holds pending tasks)
- `CELERY_RESULT_BACKEND`: Result storage (separate DB to avoid key collision with broker)

**If missing:**

- `serve` without `REDIS_URL` and `ETHERION_REDIS_URL`: API starts but rate limiting and some caches degrade gracefully
- `worker start` without `CELERY_BROKER_URL`: Worker cannot consume tasks; Celery fails to initialize
- `migrate` and `create-tenant` generally don't need Redis but may fail if credential verification requires caching

#### Authentication & Signing

```
JWT_SECRET_KEY=your-secret-here-minimum-32-characters
SECRET_KEY=another-secret-minimum-32-characters
JWT_ALGORITHM=HS256
```

**What they control:**

- `JWT_SECRET_KEY`: Signs and verifies access tokens for GraphQL and REST API calls
- `SECRET_KEY`: Signs CSRF tokens and session data (Flask/Starlette middleware)
- `JWT_ALGORITHM`: Signing algorithm (default HS256; RS256 not yet supported)

**If missing:**

- `serve` fails with: `✗ Missing required environment variable(s): JWT_SECRET_KEY`
- The platform cannot authenticate requests; all endpoints return 401 Unauthorized

**Security notes:**

- In development, use placeholder values; in production, use 32+ random bytes (e.g., `openssl rand -hex 32`)
- Rotate these keys periodically to invalidate all existing tokens
- Never commit real keys to version control

### LLM Provider Selection

```
LLM_PROVIDER=gemini
GEMINI_API_KEY=...
GOOGLE_CLOUD_PROJECT=...           # Optional: only needed when using Gemini via Vertex AI
GOOGLE_CLOUD_LOCATION=us-central1
```

**What it controls:**

- `LLM_PROVIDER`: Which LLM backend to use: `gemini`, `anthropic`, or `openrouter`
- Provider-specific credentials and project references

#### Supported Providers

**Gemini (default):**

```
LLM_PROVIDER=gemini
GEMINI_API_KEY=AIzaSy...
GOOGLE_CLOUD_PROJECT=my-project   # Optional
GOOGLE_CLOUD_LOCATION=us-central1
```

Uses Google's Gemini API directly (via GEMINI_API_KEY) or via Vertex AI when GOOGLE_CLOUD_PROJECT is set.

**Anthropic:**

```
LLM_PROVIDER=anthropic
ANTHROPIC_API_KEY=sk-ant-...
```

Uses Claude via Anthropic's API. Requires no GCP project.

**OpenRouter:**

```
LLM_PROVIDER=openrouter
OPENROUTER_API_KEY=sk-or-...
OPENROUTER_HTTP_REFERER=https://etherion.ai
OPENROUTER_APP_TITLE=Etherion
```

Uses OpenRouter's unified LLM gateway, allowing access to multiple models (Claude, GPT-4, Llama, etc.) under one API key.

#### How Provider Switching Works

The provider abstraction is implemented in `src/utils/llm_registry.py`:

1. **Provider Registry**: Each provider (Vertex, OpenAI, OpenRouter) is registered with its required environment variables and supported model tiers
2. **Lazy Loading**: Providers are imported only when needed; if `LLM_PROVIDER=anthropic` but `ANTHROPIC_API_KEY` is missing, the error occurs at first LLM call, not at startup
3. **Model Resolution**: Tiers (e.g., "pro", "flash") are resolved to concrete model names per provider; "pro" maps to `gemini-3-pro-preview` for Gemini but `gpt-4o` for OpenAI
4. **Orchestration Agnostic**: Orchestrator code in `src/services/goal_orchestrator.py` calls `get_llm(provider=...)` without knowing provider details

**Provider-agnostic code example:**

```python
from src.utils.llm_loader import get_llm

# Load the currently configured provider
llm = get_llm(provider=os.getenv("LLM_PROVIDER"), tier="pro")

# Call the LLM (same interface regardless of provider)
response = llm.invoke("What is 2+2?")
```

**Switching providers at runtime:**

To switch from Gemini to Anthropic:

```bash
export LLM_PROVIDER=anthropic
export ANTHROPIC_API_KEY=sk-ant-...
# Restart the service; next invocation uses Claude
```

No code recompilation needed; the registry dynamically loads the correct provider.

### Secrets Backend: Environment vs. Vault

```
SECRETS_BACKEND=env
```

Choose how per-tenant OAuth tokens and credentials are stored:

#### Environment Variables Backend (Default)

```
SECRETS_BACKEND=env
```

**Behavior:**

- Tenant-specific credentials (OAuth tokens, API keys for integrations) are stored in the `SecureCredential` database table
- Credentials are encrypted at rest in the database
- Best for small teams and development environments

**Use when:**

- Running on bare-metal or private cloud with direct database access
- Tenant count is low to moderate
- No external secret management infrastructure exists

**Risks:**

- Database compromise exposes all credentials
- No audit trail of secret access
- Requires secure backups of the database

#### Vault Backend (HashiCorp Vault)

```
SECRETS_BACKEND=vault
VAULT_ADDR=http://localhost:8200
VAULT_ROLE_ID=...
VAULT_SECRET_ID=...
VAULT_MOUNT=secret
```

**Behavior:**

- Tenant-specific credentials are stored in Vault under `secret/{tenant_id}/{key}`
- Vault handles encryption, rotation, and audit logging
- Best for enterprises and regulated environments

**Use when:**

- Running in a highly available, distributed setup
- Compliance or audit requirements mandate secret management
- Multiple services need access to shared secrets
- Secret rotation and versioning are critical

**Setup example:**

```bash
# 1. Start Vault
vault server -dev

# 2. Authenticate and enable secret engine
export VAULT_ADDR=http://localhost:8200
export VAULT_TOKEN=...
vault secrets enable -path=secret kv-v2

# 3. Configure Etherion
export SECRETS_BACKEND=vault
export VAULT_ROLE_ID=...
export VAULT_SECRET_ID=...

# 4. Restart API
etherion serve
```

**How secrets are read:**

When the platform needs a per-tenant OAuth token (e.g., for Gmail access):

1. Determine backend from `SECRETS_BACKEND`
2. If `env`: query `SecureCredential` table for `tenant_id`, `service_name`, and `key_type`
3. If `vault`: fetch from `VAULT_ADDR/secret/data/{tenant_id}/{service_name}/{key_type}`
4. Cache result in memory with TTL (default 5 minutes)

**Mixing backends:**

You cannot mix backends in a single deployment; choose one and stick with it. Migrating from `env` to `vault` requires:

1. Export credentials from `SecureCredential` table
2. Import into Vault at the correct paths
3. Update `.env` to `SECRETS_BACKEND=vault`
4. Restart services

### Configuration Validation: The Fail-Early System

The CLI enforces pre-flight checks before starting services. For example:

**Starting the API server:**

```bash
etherion serve
```

1. Calls `etherion/cli/cmd_serve.py:run()`
2. Calls `require_dotenv()` → if `.env` missing, print error and exit
3. Calls `check_required_vars(env, *REQUIRED_FOR_SERVE)` where `REQUIRED_FOR_SERVE = ("DATABASE_URL", "REDIS_URL", "JWT_SECRET_KEY", "SECRET_KEY")`
4. If any are empty, print list and exit
5. If all present, load and start the server

**Example error output:**

```
✗ Missing required environment variable(s):

  • JWT_SECRET_KEY
  • SECRET_KEY

  Set these in .env (or export them) before retrying.
  See .env.example for reference values.
```

This immediate, clear feedback eliminates the ambiguity of "why did the service crash on first API call?"

### Summary: Configuration Checklist

Before running any Etherion service:

- [ ] Copy `.env.example` to `.env`
- [ ] Fill in `DATABASE_URL` and `ETHERION_DATABASE_URL` (same URL ok for single-process dev)
- [ ] Fill in `REDIS_URL`, `ETHERION_REDIS_URL`, `CELERY_BROKER_URL`, `CELERY_RESULT_BACKEND`
- [ ] Generate and fill `JWT_SECRET_KEY` and `SECRET_KEY` (random, 32+ bytes each)
- [ ] Choose and configure an LLM provider (`LLM_PROVIDER`, `*_API_KEY`)
- [ ] Configure object storage (`STORAGE_BACKEND`, MinIO or S3 credentials)
- [ ] Set `ENVIRONMENT=production` if deploying to production
- [ ] Choose secrets backend: `SECRETS_BACKEND=env` (default) or `vault` with Vault credentials
- [ ] (Optional) Configure OAuth (`GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`)

Run:

```bash
etherion serve       # Validates all REQUIRED_FOR_SERVE vars; starts FastAPI
etherion worker      # Validates all REQUIRED_FOR_WORKER vars; starts Celery
etherion migrate     # Validates database connectivity; runs migrations
```

If any required variable is missing, the CLI exits immediately with a clear error. No silent failures.
